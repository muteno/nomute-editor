메시지함 실패 알림이 안 들어오던 버그 — 경로 복구

260711 · 대상: shared/msg.py · build-viewer.mjs · 워크플로 3곳

한 줄 요약 — 파이프라인(요약·분석) 실패 알림이 메시지함에 안 뜨던 원인은, 알림을 쓰는 msg.py가 gitignore된 빌드 산출물(viewer/messages.json)에 직접 써서 git add가 조용히 무시되고 있었기 때문. 알림을 git이 추적하는 입력 폴더(messages/)로 옮겨 배포까지 실제로 반영되게 고쳤어.

▍ 무슨 증상이었나

운영자가 넣은 수동 테스트 메시지(messages/260704_0725_test.md)는 메시지함에 정상으로 떴지만, 정작 자동 파이프라인이 만드는 실패 알림(fail-<base>, 인증·쿼터 실패 등)은 영영 안 들어왔음. 두 메시지가 서로 다른 경로로 viewer/messages.json에 도달하려 했는데, 한쪽만 살아있었던 것.

▍ 근본 원인

커밋 6375a3a가 메시지 시스템을 재설계하면서:

  • viewer/messages.json을 .gitignore에 넣음 (= 빌드 산출물로 규정)
  • build-viewer.mjs에 messages/*.md|json → viewer/messages.json 생성 경로를 신설
  • 그런데 shared/msg.py(실패 알림)는 안 옮김 — 여전히 viewer/messages.json에 직접 씀

결과: 러너에서 msg.py가 알림 파일을 만들어도, 워크플로의 git add viewer/messages.json이 gitignore 규칙에 걸려 무시됨. 실측 재현:

✗git add viewer/messages.json → "The following paths are ignored by one of your .gitignore files ... Use -f" → 스테이징 안 됨

게다가 설령 커밋됐어도, 다음 배포 빌드가 messages/*.md에서 재생성하며 덮어써 이중으로 유실될 구조였음. (워크플로 주석엔 2>/dev/null || true로 에러를 삼켜서 "있을 때만 커밋"이라 적혀 있었지만, 실제로는 있어도 무시되고 있었다.)

▍ 바뀐 파일

  • shared/msg.py — viewer/messages.json 직접쓰기 → messages/<id>.json set/clear + 24h prune (파일명 sanitize)
  • build-viewer.mjs — 합성 시 t(만료 ms)·level(warn) 필드 보존 + 정렬을 시각 정규화(msgMs)로
  • .github/workflows/news-analyze.yml · news-ask.yml — git add viewer/messages.json → git add -A messages
  • .github/scripts/cardmake.sh — 동일 교정

장점

  • 실패 알림이 실제로 배포·표시됨 (버그 해소)
  • 입력=messages/(추적) / 산출물=messages.json(gitignore) 단일 원칙으로 정합
  • 수동 메시지와 자동 알림이 한 경로로 통일
  • t·level·정렬까지 뷰어 로직과 일관

단점 / 트레이드오프

  • 알림 1건 = messages/에 파일 1개 커밋 (봇 커밋에 섞임 — append라 무해)
  • id별 파일명 sanitize에 의존 (경로 안전화)
  • 배포 빌드 1사이클 뒤 반영 (즉시 아님 — 기존 파이프라인과 동일 특성)
