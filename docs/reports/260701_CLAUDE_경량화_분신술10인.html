1운영자가 꼭 알아야 할 3대 반전

① @import로 쪼개면 토큰은 0 줄어든다 기술 확인

Claude Code 공식 문서 verbatim: "imported files are expanded and loaded into context at launch … does not reduce context." 파일을 @경로로 나눠도 세션 시작에 전부 합쳐 로드된다. 더 나아가 물리 분할(CLAUDE-part2.md)은 auto-load가 하위 md를 안 합쳐서 규칙이 통째 증발한다(9번·5번 일치). ∴ 유효 감량 = 물리적 삭제·압축·이관뿐.

② 최대 출혈은 --bare가 이미 막았다 — 남은 이득처가 다르다 비용 지도

전체 LLM 호출의 81%인 judge(gate·breaking)는 --bare로 라우터를 이미 안 읽는다(감량 이득 0). 남은 로드처 = ⑴ 인터랙티브 세션 1회/세션 ⑵ 생성경로(analyze·cardmake·ask·revise·kmake·lymake). 생성경로는 도구 의존 때문에 --bare가 금지돼 있어 — 라우터를 물리적으로 줄이는 것이 이 경로 비용을 깎는 유일한 레버다. 그중 card(24h 44콜·캐시적중 낮음)가 ROI 최고.

③ 무게의 근본원인 = SSOT 이중화 + 이력 서사 진단

상세의 정본이 이미 docs/·스킬에 존재하는데 라우터가 원문급으로 복제 보유 중이다 — docs/news-pipeline.md·docs/curation-algorithm.md(뉴스큐), nomute-design 스킬·docs/CII…md(디자인), docs/작업이력.md(완료 로그). 여기에 날짜 스탬프 135개·⚠️ 79개·취소선 26개·최장 단일 불릿 4,295자가 규칙과 뒤엉켰다. 중복분을 포인터로 치환 + 이력을 원장으로 이관 = 품질 손실 0의 최대 절감.

❌ 3단계 · 비권장 (물리 분할)

라우터를 별도 md로 쪼개기 — 기술적 고장

auto-load가 하위 md를 안 합치므로 CLAUDE-part2.md의 규칙은 로드 안 됨 = 증발(5번·9번 확정). @import도 토큰 감량 0. 대안=.claude/rules/+paths는 공식 조건부-로드 기능이나, "항상 로드" 룰엔 부적합 → 앱별·파일타입별 국소 룰(예 viewer/** 디자인)만 이관 가능. 하려면 §기틀보호+Opus 5명+§수정모드 전체.

5💡 아이데이션 베스트픽

지금 당장: 1단계(이력 이관 + 문체 압축)부터

무게의 최대 덩어리(SSOT 이중화·이력 서사)를 최소 리스크로 걷어낸다. docs/작업이력.md(이관처)·nomute-design 스킬·docs/CII…md가 이미 갖춰져 있어 새 문서 창작이 거의 없다. 이건 §🧪 제외항목(문구·케이스로그)이라 기틀 절차 면제·자가검증만으로 오늘 안전하게 가능.

병행: 감량을 "기계로 안전하게" 만드는 가드레일 3종

  • check_refs.check_alwaysload() 신설 — "항상 로드" 절의 삭제·이동을 커밋 전 차단(rc=1)
  • 물리분할 봉쇄 — CLAUDE*.md가 2개 이상이면 rc=1(증발 사고 원천 차단)
  • 구조/문구 분리 강제 — diff가 절 헤더(## )를 건드리면 자동 "기틀 변경"으로 분류 → Opus 5명 게이트

이러면 "표현 압축은 자유, 구조 변경은 게이트"가 소프트룰이 아니라 기계 계약이 된다 — 적대자(9번)의 핵심 우려를 해소.
