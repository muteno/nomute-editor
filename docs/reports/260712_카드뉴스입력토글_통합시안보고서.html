3. 리스크 · 장단점 · 새 아이데이션

리스크 (적대적 감사)

  • 회귀① 오버레이 발사 페이로드가 입력 방식(textarea/칩)에 따라 갈림 → 두 경로가 동일한 *별표* 문자열로 수렴하지 않으면 백엔드(nomute_*.py) 렌더가 어긋남. 방어 = 칩 → *강조* 직렬화 단일 함수로 통일.
  • 회귀② card_news.py는 절대규칙: 수정 금지·import만. 게다가 이중소비 — gen_cards.py·recompose_card.py·cardmake.sh(라이브 뉴스 카드 파이프)도 같은 스크립트 import. 한 줄 고치면 /comp+뉴스 카드 슛/edit 동반 사망. 방어 = 입력 UI만 얹고 generate() 시그니처 무접촉.
  • 회귀③ ⚠핵심 엔드포인트 상이 — 썸네일 app2 = /api/thumb(오버레이 nomute_compose.py) vs 카드뉴스 app5 = /api/compose(card_news.py 1080×1350 그라데이션). 다른 렌더 엔진. 토글은 입력 모드만 바꾸고 발사 경로는 각자 유지 — "렌더 통합"으로 오해하면 §기틀 위반. 저장모델도 상이(compose=git 커밋 / thumb=R2).
  • 회귀④ 쌍둥이 드리프트 — 칩 로직은 thumb #cardBox ↔ index #edChips(카드 변경) 동기화 의무(§디자인 통일 기틀). 칩 로직 수정 시 두 파일 동시 반영 필수, 복제 금지 → DOM 이동 공유(저작권 #cpBlock 선례).
  • 회귀⑤ 토글 상태가 snapForm/IndexedDB 복원에 안 들어가면 새로고침 시 입력 방식·강조 유실. 토글 OFF가 edLines 리셋 금지(draft 왕복). 방어 = 스냅샷 키에 cnMode+칩 배열 포함.
  • 회귀⑥ 신규 토글 무단 배치 = §디자인 j) 위반(260703 국제칩 사고 재발). 방어 = 이 시안 승인 후에만 머지. 제미나이-0 이중잠금(Lock A/B)도 통합 워크플로에 GDRIVE 유입 절대 금지.
  • 회귀⑦ 칩 backdrop-filter 재샘플 flicker(모달 내 타이핑 번쩍). 방어 = 카드뉴스 정본이 이미 flicker-safe(dashed border·불투명) → 그대로 계승(신규 글래스 backdrop-filter=0).

장점 / 단점

  • 장점 별표 문법 암기 불필요 — 단어 탭으로 강조를 보며 지정(비전문가 보정).
  • 장점 카드뉴스 탭과 동일 입력 UX → 학습곡선 0, 두 탭 오가도 일관.
  • 장점 컴포넌트 전부 계승(.onoff+.ed-chips) = 신규 CSS·토큰 0, 드리프트 0.
  • 장점 폭 초과·자간 검증을 칩 단위로 실시간(카드뉴스 로직 재사용).
  • 단점 폼 복잡도 ↑(토글 1개+분기) — 어포던스로 흡수(산문 안내 없이 토글 색만).
  • 단점 칩 형태소 분할이 긴 문장에서 칩 수 폭증 → 모바일 스크롤. 카드뉴스와 동일 하드캡(줄 수) 적용.

새 아이데이션 (베스트 픽)

  • A안 ★추천 토글만 최소 추가 — 오버레이 섹션 .csec 우측에 .onoff "카드뉴스" 토글. ON=칩 에디터(#cardBox의 edChips 로직 재사용) / OFF=기존 #cLines. 발사 시 칩→*별표* 직렬화로 백엔드 무접촉. 10분·단기 안전 산출, 기틀 리스크 최소.
  • B안 두 입력을 양방향 실시간 미러(textarea ↔ 칩) — 어느 쪽을 만져도 반대편 동기. 강력하나 desync 버그면 데이터 손실 위험 → v2 후속.
  • C안 단일 소스 통합 — 썸네일·카드뉴스가 edLines 한 상태를 공유해 "동일 내용이 통합 메뉴에 붙는다"를 문자 그대로 구현(한 번 입력 → 두 산출물). 가장 이상적이나 상태 격리(PIMG desync 선례) 재설계 필요 = 기틀 → 별도 승인.
  • 덤 강조 색은 콘텐츠 상수 #0FFD02 형광그린 유지(§핵심명령 3-b-1 — UI 팔레트 개편 비동행). 칩 .on 미리보기색(accent 터쿼이즈)과 결과물색(형광그린)이 다른 점을 각주로 명시.
