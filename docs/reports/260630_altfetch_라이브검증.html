260630 · RSS 자동수집 경로(to_pending) alt-fetch 이식 · 자동수집 1회 테스트로 실증 · 머지·검증 완료

한 줄 요약

RSS로 자동 수집된 차단매체(조선·동아 등) 기사도, 이제 같은 사건을 보도한 다른 매체로 본문을 자동 회수해서 요약이 깨지지 않는다. 라이브에서 chosun 0바이트(차단) → SBS 1850바이트 회수로 실증됨.

1. 무엇이 문제였나 배경

전 — RSS 자동수집의 빈틈

BEFORE

한국 언론사 본문 페이지는 GitHub Actions(데이터센터 IP)에서 긁으면 403으로 막힌다(조선·동아·연합·중앙·한겨레 등 WAF 차단).

운영자가 폰으로 직접 고른 기사(픽)·자동속보픽은 이미 # alt:(같은 사건 다른 매체)를 심어 우회했는데, RSS 자동수집 경로(to_pending)만 그걸 안 심었다.

→ 자동수집된 차단매체 기사는 본문이 비어 요약 실패/빈약.

후 — 픽 경로와 통일

AFTER

to_pending.py에 load_alts() 신설 — 수집기가 만든 articles.json의 같은 사건 클러스터(cluster_members)를 pending 파일에 # alt:로 심는다.

엔진(analyze.sh)·데이터(클러스터)는 원래 있었음 — 그저 배선만 연결. 큐레이션 알고리즘 0 변경.

검증패턴은 픽 경로 재사용: http(s) 도메인만·자기 제외·1500자 cap·공백/개행 멤버 거부(마커 주입 차단).

2. 라이브 end-to-end 실증 스모킹건

자동수집 1회 테스트(scrape analyze=true top=3)로 진짜 차단매체 기사를 돌렸다 — chosun 부동산 기사("동탄 22억…칼 빼든 정부").

1
수집·적재 — scrape #1423이 chosun 기사에 # alt: 26개 매체 심음 (SBS·한겨레·한경·경향·mk·뉴시스…). 01·03 기사는 클러스터 단독→url만(하위호환).
2
원매체 fetch — analyze가 chosun.com 본문 fetch → 0바이트(데이터센터 IP 403 차단 확인).
3
alt-fetch 발동 — 대체매체 루프가 SBS에서 1850바이트 본문 회수:
01:57:52 원매체 본문 빈약(0B<900) → 더 완전한 대체매체 채택: https://news.sbs.co.kr/news/endPage.do?news_id=N1008633510… (1850B) 02:09:11 검색 유사 보강 — alt_urls 주입(26개 매체) 02:09:11 성공 → queue/260630-1047-ER3BHKVE65ELPLYBDTCAP3H2.md

결과 — ANALYSIS_FAILED 없이 완전 다이제스트 생성(상세 수치·사실 다 채움). 출처 = SBS·헤럴드경제·뉴시스 교차확인. alt 없었으면 0바이트→분석실패였을 기사.

4. 장단점 · 효율 · 품질 리스크

👍 장점

  • 회수율 자동수집 차단매체도 본문 회수(픽 경로 89%급).
  • 무료 새 deps 0 · 외부 도구 0 · 기존 엔진·데이터 재사용.
  • 기틀무손상 큐레이션 알고리즘·신호·랭킹 0 변경 = 단서 연결만.
  • 통일 픽·자동속보픽·RSS자동 = 세 입구 모두 동일 alt 로직.

⚠️ 단점 · 한계

  • 조건부 즉효는 자동수집(analyze=true) 켤 때만 — 현재 OFF(미래 대비).
  • 잔존 클러스터 단독(대체매체 0)인 차단기사는 여전히 회수 불가 → 폰 paste/Tailscale로 우회.
  • 비용노브 자동수집 ON 시 과금은 alt 아닌 자동 요약+썸네일에서 발생(top·MAX_BATCH로 상한).

효율성

alt-fetch 루프는 원매체 본문이 빈약할 때만 작동(cur_len < 900B 게이트)·충분한 본문 확보 시 조기 종료(불필요 fetch 차단). 정상 매체는 추가 비용 0.

품질저하 리스크

≈0. 기존 동작은 그대로(alt 비면 url만 = 폰공유와 1바이트도 안 다름)·신규는 차단 케이스만 개선. 보안 = 공백/개행 멤버 거부로 가짜 마커(# body:) 주입 차단(자가검증 중 발견·수정).
