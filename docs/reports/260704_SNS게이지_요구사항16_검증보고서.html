01 SNS 수집 글에 편향 매기기 — 글 읽을 수 있는지부터 확인 완료
“SNS 수집한 다음에 편향도를 각각 매겨줄수있어? 글을 읽을수있는지부터 확인해보자”
🔄기존 → 현재
기존: SNS 탭은 제목만 수집(title·url·sources·burst)·편향 개념 0 → 현재: 글 읽기 5단계 실측(커뮤=이슈링크 목록만·네이버=PostView 본문 O) 후 '출처 성향'으로 방향 확정
⚠️리스크
커뮤 본문 미접근 = 글 내용 기반 편향은 원천 불가. 소재가 비정치(POLITICS 컷)라 정치편향 대상 자체가 희소
⚖️장단점
장: 실측으로 헛된 LLM 편향 파이프를 사전 차단·토큰0 / 단: 글 내용이 아닌 '출처' 프록시
🛠단점 보완
출처 성향맵(SOC_LEAN_MAP) 정확도로 프록시 품질을 보완
📅추후 계획
네이버 본문(PostView 추출됨) 한정으로는 뉴스식 LLM 내용편향 병행 여지
💡아이데이션
커뮤 원본 직접 크롤(WAF 403 리스크)·이슈링크 listview 파싱 심화
♟신의 한수
'글 읽을 수 있나부터' 요청대로 5단계 실측 먼저 → 엉뚱한 LLM 편향 파이프 안 짓고 방향 전환
❓이건 모르겠다
커뮤 본문 우회(데이터센터 IP 차단) 접근의 실효성
02 뉴스 요약과 동일 방식(보정 없이 판단만) + 메뉴 표시도 비슷하게 완료
“뉴스 요약하고 동일함. 보정만 안하고, 판단만. 메뉴 내 표시 방법도 비슷하게 하고자 함”
🔄기존 → 현재
기존: 뉴스=LLM이 본문 읽고 bias N/10 판단+약보수 보정 재작성 → 현재: SNS는 소재·본문 제약으로 글내용 판단 부적합 → '표시'만 뉴스 편향바(--bias-* 토큰·5단 색) 계승
⚠️리스크
'뉴스와 동일'을 글내용 LLM 판단으로 오해하면 전건 N/A. 표시 축만 계승됨
⚖️장단점
장: 뉴스 편향바 토큰·진보~보수 축 그대로 계승=학습 전이·일관 / 단: 판단 주체가 LLM(뉴스)→규칙(SNS)로 상이
🛠단점 보완
표시 축(진보↔보수)을 동일 유지해 사용자 인지 부담 최소화
📅추후 계획
네이버 본문 있는 이슈는 뉴스식 판단 병행 옵션
💡아이데이션
게이지에 '출처 기반' 툴팁으로 판단 근거 명시
♟신의 한수
'보정 안 함'이 SNS엔 '원글 재작성 없음'으로 자연 대응(SNS는 남의 글이라 못 고침)
❓이건 모르겠다
운영자가 기대한 '동일'의 정확한 경계(표시만인지 판단로직까지인지)
03 어느 성향 커뮤에서 나온 글인지(출처 성향)만 확인 완료
“1,2 ㄱㄱ 어느쪽 성향 커뮤에서 나오는 글인지만 확인하면됨”
🔄기존 → 현재
기존: 글 내용 LLM 편향(N/A 다발) → 현재: 각 이슈의 sources[]를 커뮤 성향맵으로 매핑=출처 성향(LLM·본문·쿼터 0)
⚠️리스크
커뮤 성향맵이 통념 근사라 논쟁적. 성향 미등록 커뮤는 게이지서 제외
⚖️장단점
장: 토큰0·소재 비정치여도 작동·즉시 계산 / 단: 커뮤 성향이라는 프록시의 부정확성
🛠단점 보완
성향맵을 웹검색 근거로 지속 보정(운영자 당일 8회 검색 반영)
📅추후 계획
커뮤 성향 재검토를 별건 세션으로
💡아이데이션
커뮤별 성향을 커뮤니티 자체 데이터로 실증(현재는 통념)
♟신의 한수
글내용 LLM 대신 '출처'로 프레임 전환 = 문제(본문없음·비정치)를 우회하고 즉시 실현
❓이건 모르겠다
특정 커뮤(인벤 중립 등) 성향 분류의 타당성
