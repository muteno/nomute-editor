자막 생성기 결합 평가 — 상용 대비 18개 인덱스 + 즉시 개선

260707 · 실측 = 결합 헤드리스 렌더 + E2E 2회 · 벤치 = CapCut·Vrew·Submagic·Zapcap·Captions.ai(분신술 R1)

총평 — 결합 상태 실측 소견

뷰어 하단 + → 영상 → 자막 생성기 결합 자체는 성공적이다. 모달 셸(타이틀 고정·세부 3탭)과 자막 생성기 폼이 한 화면에 들어오고(문서 높이 619px = 스크롤 부담 낮음), 소머리·칩·눌림·글래스가 전부 기틀 계승이라 다른 도구(이미지·영상 프롬프트)와 시각 언어가 일치한다. 진입 마찰도 + 탭 → 영상 탭 = 2탭으로, CapCut의 "영상 선택 → 오토캡션" 2~3탭과 대등하다. 문제는 결합의 겉이 아니라 설정을 고르는 순간의 체감이었다 — 아래 18개 인덱스에서 상용과 가장 벌어졌던 5개(프리셋 부재·미리보기 부재·재작업 비용·설정 위계·좁은 폰 밀림)를 이번에 바로 개선했고, 나머지는 로드맵과 의도적 수용으로 갈랐다.

결합 UX 정밀 소견 (실측 5장면)

장면 1 — 진입. 하단 네비 가운데 + 를 누르면 라디얼 3항목(뉴스 요약·이미지·영상)이 펼쳐지고, 영상을 고르면 모달이 뜨면서 자막 생성기가 기본 활성 탭으로 로드된다. 실측에서 + 탭부터 폼 조작 가능까지 약 3초(iframe 로드 포함·캐시버스트 상태)였다. 상용 앱의 "앱 실행 → 프로젝트 생성 → 영상 임포트 → 캡션 메뉴 진입" 3~4단계와 비교하면, 이미 뉴스 뷰어 안에 있는 운영자 기준 실질 마찰은 더 낮다. 다만 상용은 영상 선택이 진입과 동시인 반면 우리는 진입 후 URL/파일을 고르는 순서라, "영상부터 던지는" 멘털 모델과는 한 단계 어긋난다 — 이건 도구가 다목적 입력(URL·파일·SRT 텍스트)을 받는 구조 대가라 수용이 맞다.

장면 2 — 설정 탐색. 개선 전 화면은 소머리 2개 아래 칩 15개가 동일한 시각 무게로 깔려 있었다. 파워유저(운영자)는 축이 눈에 들어오지만, 첫 판단 비용이 "이 6축이 뭘 하는지"를 전부 읽는 것부터 시작한다. 상용이 전부 프리셋 카드로 진입하는 이유가 이거다 — 선택지를 고르는 게 아니라 무드를 고르게 한다. 이번 개선에서 빠른 프리셋 행을 소머리 바로 아래 첫 행으로 올려 같은 문법을 얹었고, 프리셋과 칩이 실시간 동기라(6키 일치 판정) "프리셋으로 시작해 칩으로 미세조정"이 자연 동선이 됐다.

장면 3 — 설정 확신. 개선 전엔 모양(기본/깔끔/박스)이라는 낱말만 보고 결과를 상상해야 했다. 이번 미리보기 바는 모양·크기·키워드 그린·카라오케 스윕을 즉시 보여준다. 실측 스크린샷에서 박스 프리셋을 누른 순간 반투명 상자·"강조" 그린·스윕 채움 중간 프레임이 그대로 잡혔다. 단, 이건 CSS 근사다 — 실번인(libass)과 폰트 힌팅·줄바꿈 지점이 미세하게 다르니 픽셀 확인 도구가 아니라 방향 확인 도구로 읽어야 한다(각주를 바 자체엔 안 넣고 여기 남긴다 — 화면 산문 금지 원칙).

장면 4 — 대기. 발사 후 받아쓴 자막(전사)이 약 1.5~2분에 먼저 열리고(조기 커밋 실측 89초 선례), 완성 자막·완성 영상이 뒤따른다. 이 "먼저 편집하게 해주는" 흐름은 상용에도 드문 정직한 진행 설계라 유지 가치가 크다. 반면 총 대기(콜드런 최대 8분)는 온디바이스 상용과 비교 불가한 구조 격차 — 대신 반복 구간(스타일 변경)을 다시 입히기로 1분급으로 끊어낸 게 이번의 실질적 답이다.

장면 5 — 회수. 완성 영상은 모달 안 플레이어로 바로 재생되고, 다운로드는 같은 출처 프록시라 iOS에서도 제스처가 살아 저장이 된다. 모달을 최소화해도 폼과 결과가 보존되고(도구별 iframe 보존 구조), 영상 프롬프트 탭으로 넘어갔다 와도 리로드가 없다. 이 결합 특성(멀티 도구 동시 최소화)은 상용 단일 앱엔 없는 뷰어만의 이점이다.

아키텍처 평가 — 상용 클라우드 vs 노뮤트 Actions 파이프

상용 자막 툴의 본체는 "전용 GPU 서버 + 실시간 렌더 미리보기"고, 구독료가 그 서버비다. 노뮤트 파이프는 무료 GitHub Actions 러너 + 구독 OAuth LLM + R2로 같은 산출물(자막 텍스트·SRT·번인 영상)을 만든다. 이 구조 선택의 결과를 정직하게 갈라 보면 이렇다.

구조가 이기는 곳: ① 비용 — 추가 지출 0(러너 분·R2 프리티어·기존 Claude 구독). 상용 $20/월급 기능이 무료로 돈다. ② 의역 품질 — 상용 기계번역과 opus 4.8 감성 의역의 격차는 명확하다. 릴스 문법(짧은 조각·구어체·펀치)을 지침으로 강제할 수 있는 것도 LLM 파이프만의 것. ③ 재현성·감사 — 모든 산출이 git에 남고(subs.md·subs.json·video.json), 실패가 error.log로 표면화된다. 상용은 블랙박스다. ④ 확장 — 이번 다시 입히기처럼 파이프 조각을 재조합하는 개조가 워크플로 수십 줄로 끝난다.

구조가 지는 곳: ① 지연 — 러너 부팅+환경 복원이 기본 1~2분, 콜드런 whisper 다운로드가 최대 수 분. 실시간성은 영원히 못 이긴다. ② 미리보기 충실도 — 서버 렌더 프리뷰가 없어 CSS 근사로 때운다(이번 개선의 한계 그대로). ③ 동시성 — concurrency 직렬화라 연속 발사가 줄을 선다(운영자 1인 도구라 실질 무해). 판정: 운영자 1인·비용 0·품질 우선이라는 노뮤트 전제에서 이 구조는 옳다. 격차는 "실시간이 필요한 순간"에만 아프고, 그 순간의 대부분(스타일 반복)을 이번에 1분급으로 줄였다.

툴별 정밀 비교 — 각자 뭘 잘하고, 우리와 어디서 갈라지나

CapCut(오토캡션): 최강점은 속도와 템플릿 생태계다. 온디바이스 STT라 1분 영상이 수십 초에 끝나고, 카라오케·팝·바운스·타자기 등 모션 템플릿 수십 종이 원탭이다. 대신 한국어 인식률이 Whisper large 대비 들쭉하고, 번역 자막은 기계번역 직역체라 릴스 톤을 살리려면 결국 손으로 고쳐야 한다. 우리는 속도에서 지고 의역 품질에서 이긴다 — 그리고 CapCut의 글자단위 모션은 벤치 판정대로 libass 번인에선 비용 대비 효과가 없어 카라오케 하나로 끊은 게 맞다.

Vrew: 편집기가 본체다. 전사 텍스트를 문서처럼 고치면 영상이 따라 잘리는 문법(텍스트 기반 편집)이 강력하고, 무음 구간 일괄 삭제·검색 치환이 좋다. 우리 상세 편집기는 단어 빼기·수정·분열·병합·찾아 바꾸기·군더더기 원클릭까지 Vrew 문법의 핵심을 이미 이식했고, 컷 스킵 미리보기도 있다. 갈라지는 지점은 딱 하나 — Vrew는 삭제가 실제 영상 컷으로 이어지고 우리는 아직 미리보기만 건너뛴다. 그래서 로드맵 P1이 무음/필러 영상 컷이다.

Submagic·Zapcap: 쇼츠 특화 자동화(키워드 강조색·이모지·필러 제거·트림)를 API로 파는 물건들이다. 문법 자체는 우리 파이프와 가장 닮았는데, 결정적 차이는 손맛의 방향 — 걔들은 "자동 제안 → 사람이 승인" 루프고, 우리는 "LLM이 지침대로 한 번에 → 사람이 편집기로 다듬기" 루프다. 키워드 강조를 예로 들면 Submagic은 색 3개를 수동 배정하고, 우리는 opus가 조각마다 핵심어를 별표로 자동 태깅해 콘텐츠 그린 하나로 통일한다. 브랜드 일관성 관점에선 우리 쪽이 오히려 프로덕션에 가깝다.

Captions.ai: AI 아이컨택 보정·더빙 등 자막 밖 기능이 본체라 직접 비교 대상은 아니다. 자막 축만 보면 위 셋의 부분집합.

종합: "STT 정확도 × 의역 품질 × 스타일 시스템 × 반복 속도" 네 축에서 우리는 1·2축 우위, 3축은 이번 개선으로 근접(프리셋+미리보기+축별 칩), 4축은 다시 입히기로 반복 구간만 상용급. 남은 열세는 초회 대기(구조 비용)와 무음 컷(P1)이다.

개선 후: 빠른 프리셋 행 + 위치/크기 행 분리 + 실시간 미리보기(박스·키워드 그린·카라오케 스윕)

로드맵 — 남은 격차의 실행 순서와 이유

P1 · 무음/필러 영상 컷(인덱스 12): 남은 격차 중 유일하게 "산출물 품질"을 직접 올리는 것. 지금도 편집기에서 조각을 삭제하면 미리보기는 건너뛰지만 실제 영상은 그대로다. 편집기의 삭제 조각 s/e를 그대로 ffmpeg select/concat에 넘기면 번인 파이프를 재사용해 "군더더기 구간이 실제로 잘린" 완성 영상이 된다. 다시 입히기 배선(subs 재사용·reburn 모드)이 이번에 생겨서, 이 확장은 "편집 상태를 서버로 보내는 입력 하나 + ffmpeg 필터 조립"만 남았다. 체감 = Submagic Trim 급.

P2 · 폰트 2~3종(8): 러너 apt에 폰트 팩 추가 + 모양 칩 옆 폰트 칩. 눈누 상업 무료(예: 프리텐다드·검은고딕류)로 브랜드 안 깨지는 선택지만. P2 · 공유 시트(17): navigator.share에 파일 넘기는 한 줄 — iOS/안드로이드에서 다운로드 없이 인스타 직행. P2 · 키워드 다색/수동(7)·타임라인 드래그(10)·브랜드킷(9): 수요가 확인되면. 특히 9는 콘텐츠 색 축이라 운영자 색 지정이 선행되어야 한다.

안 하는 것(재확인): 이모지 자동(13)·화자 자동(14)·다국어 출력(15) — 벤치 판정과 노뮤트 타깃(한국어 릴스·1인 운영) 기준 과설계. 이 선긋기 자체가 상용 대비 우리 도구의 단순함을 지키는 결정이다.

운영자 사용 시나리오 3종 — 개선 후 실제 동선

시나리오 A(가장 흔함 — 외신 릴스 재가공): 릴스 URL 붙여넣기 → 빠른 프리셋 [릴스 기본] 확인 → 자막 생성. 2분쯤 뒤 받아쓴 자막이 먼저 떠서 군더더기 원클릭·오타 수정 → 완성 자막과 번인 영상 도착 → 다운로드. 총 조작 4~6탭, 타이핑 0.

시나리오 B(스타일 고민): A와 같게 만든 뒤 결과가 심심하면 — [정보 박스] 프리셋 탭 → 미리보기 바로 확인 → 다시 입히기. 1분 뒤 같은 의역으로 스타일만 바뀐 영상. 예전엔 이 반복마다 전체 재생성(3~8분+쿼터)이었다.

시나리오 C(폰 촬영 원본): 20MB 이하 영상 파일 첨부(자막 붙이기 ON이면 원본 그대로 업로드) → 생성. 업로드는 일회용 브랜치로 가서 레포 비대 0, 처리 후 자동 소멸. 20MB 초과면 음성만 추출돼 자막 텍스트로 받고, 합성이 필요하면 URL 경로 안내가 뜬다.

세 동선 모두에서 "프롬프팅"은 존재하지 않는다 — 운영자가 고르는 건 무드(프리셋)와 예외(칩)뿐이고, 지침·옵션 문장 조립은 전부 기계가 한다. 이게 이번 결합·개선의 목표였고, 실측 기준 달성이다.

결론

결합은 구조적으로 성공했고(기틀 계승·2탭 진입·멀티 도구 공존), 상용 대비 실질 격차는 "고르는 순간의 체감" 5개와 "반복 비용" 1개였다 — 6개 전부 이번에 코드로 닫았다. 남은 것 중 가치가 큰 건 무음 컷(P1) 하나고, 나머지는 P2거나 의도적 수용이다. 강점(opus 의역·비용 0·정직한 진행·git 감사 가능성)은 상용이 못 따라오는 축이라, "실시간 미리보기 완전체"를 쫓기보다 P1과 콘텐츠 품질(의역 지침 고도화)에 힘을 싣는 게 맞다.
