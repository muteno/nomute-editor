5. 장단점 · 효율 · 리스크

✅ 장점

  • 일시 과부하에 기사를 안 잃음 — 짧은 깜빡임은 인라인서 즉시 흡수, 35분 윈도는 sweep로 자가치유
  • 뷰어가 진실과 동기화 — 재시도는 앰버, 진짜 실패만 빨강(오해 제거)
  • 게이트웨이 5xx 변종까지 커버
  • 추가형(additive) — 3단계 파이프라인 기틀·제미나이0 잠금 불변

⚙️ 효율

  • 529 = 토큰 과금 0(서버가 생성 전 거부) → 쿼터 영향 무시 가능
  • 인라인 백오프 ~45s/건 · 잡 타임아웃 90분에 한참 못 미침
  • 최악 호출 = 1기사 15회(전부 0토큰 fast-fail) · 캡 5로 유한

⚠️ 품질저하 리스크 (전부 비차단·수용)

  • 장기 과부하 시 자동 재시도가 "즉시"가 아니라 최대 ~20분(sweep 주기) — 단 인라인 3회가 짧은 깜빡임은 즉시 흡수. 칩 툴팁이 '자동 재시도 대기'로 기대치 동기화.
  • 1시간+ 지속 과부하는 캡(5회) 도달 전까지 GitHub 알림엔 안 뜸(뷰어 칩으로만) — 자가치유 노이즈 억제와의 의도된 트레이드오프.
  • 드문 FP: 기사 본문 앞 8줄에 영어 "Service Unavailable" 등을 그대로 인용 시 1회 헛재시도(무해·곧 수렴). 숫자 '500명·503호'는 앵커로 차단됨.
