③ 장단점 · 효율 · 품질저하 리스크

  • 일관성·계승: 첨부 슬롯·토글·픽토그램 전부 기존 컴포넌트(오버레이 첨부·.geni-opt·SVG) 계승 = raw 창작 0, 토큰 게이트 통과.
  • 무회귀 설계: 참고 이미지는 가산적 — 미첨부면 발사·렌더가 바이트 동일. R2·업로드 인프라 무접촉(base64 dispatch 경유).
  • 효율: PNG 단일화로 무의미한 선택 제거 · postMessage에서 48KB base64 제외(브로드캐스트 경량화).
  • 참고 이미지 512px 상한: dispatch 입력 한도(65535자) 때문에 참고 이미지는 512px로 축소 전송 → 미세 디테일 참고엔 충분하나 초고해상 원본 보존은 아님(참고/재구성 용도라 적정).
  • 라이브 렌더 미검증: 세션이 Cloudflare Access 뒤라 실제 Gemini 렌더는 못 돌림 — 서버 로직은 런타임 실증했으나, 머지 후 폰에서 실제 첨부→생성 1회 확인 필요.
  • 원본유지 시맨틱: 둘 다 이미지를 Gemini에 전달·프롬프트 지시만 분기하는 v1 해석. "원본 그대로"가 "생성 없이 원본 그대로 쓰기"였다면 별도 조정 필요.
