대용량 업로드(R2 직업로드 ≤2GB) + 편집기·변환 4K 지원

260711 · PR #1941 · 운영자 확정(풀세트) — 30MB 병목 해소 + 생성 4K ↔ 후편집 1080p 엇박 해소 + 침묵 다운스케일 표면화

1무엇이 바뀌나 (쉬운 설명)

① 편집기·변환 탭의 파일 업로드가 30MB → 2GB로 커진다. 파일이 GitHub를 거치지 않고 저장소(R2)로 조각조각(32MB) 직행하는 방식이라 사실상 용량 제한이 사라지고, 실질 제한은 원래 있던 길이 캡(편집기 10분)만 남는다.

② 편집기 해상도 카드에 원본(4K) 선택지가 생긴다. 이걸 고르면 4K가 4K로 나온다(지금까지는 무조건 1080p급으로 축소). 변환 탭은 해상도 '원본'이면 4K가 그대로 통과한다(지금까지는 4K 입력 자체를 거절).

③ 4K를 선택하지 않아 1080p로 줄여 내보낼 때, 이제 결과에 "원본 3840×2160 → 긴 변 1920 축소" 안내(note)가 뜬다 — 말없이 줄이던 것 교정.

폰/PC 파일(≤2GB)→ api/upload · 32MB 조각→ R2 up_src/* (일회용)→ 러너 aws s3 cp 회수→ ffmpeg 편집/변환→ 결과 + 원본 자동 삭제

3장단점 · 효율성

장점대용량·4K 파일이 드디어 1급 입력. GitHub 히스토리 비대 0(대용량은 git 자체를 안 거침). 업로드 진행%가 보임. 침묵 다운스케일 폐지 = 정직성↑.

효율비용 축 = R2 저장(일회용·잡 끝 삭제 = 잔존 0) + Actions 분. 4K 잡은 1080p 대비 인코딩 시간 약 3~4배라 길이 캡으로 예산을 묶었다(편집기 3분·변환 2분 — 릴스/쇼츠 용도엔 충분). LLM 토큰 0(순수 ffmpeg 경로 불변).

트레이드오프4K 잡 대기 시간 체감↑(편집기 최악 ~40분 백스톱 · 완료 웹푸시가 커버). 60fps 보간은 4K에서 미지원(예산 4배 폭발 — 1080p로 내리면 가능). 변환 URL 소스는 1920 유지(4K는 파일 업로드로).

4품질 저하 리스크 (정직)

  • 기존 경로 회귀 0 설계 — R2 미바인딩(현 라이브 상태)이면 모든 동작이 종전과 동일. 편집 축 결측 = 순수 자막 경로 캡(폭 1080)도 불변(축소 note 한 줄만 추가).
  • 4K 인코딩 품질 — 출력은 동일 crf(편집기 20 · 변환 19) H.264라 픽셀당 품질 기준 동일. HDR(10비트) 원본은 종전과 같이 8비트 SDR 변환(톤매핑 없음 = 색 물빠짐 가능 · 기존 한계 유지).
  • 미완 멀티파트 잔재 — 업로드 중단 시 abort를 쏘지만 fire-and-forget. 잔재는 무해(과금 미미)·후속으로 R2 수명규칙 1일 권장.
  • 러너 예산 최악 스택 — 4K+배경음 제거 동시는 컴포즈 60분 캡에 근접 가능 → 초과 시 error.log 정직 실패(무한 행 없음).

6운영자 액션 — R2 바인딩 1회 (이거 해야 2GB가 켜짐)

Cloudflare 대시보드 → Workers & Pages → nomute-editor(Pages) → Settings → Bindings → Add → R2 bucket → 변수명(Variable name) = R2 · 버킷 = 러너 시크릿 R2_BUCKET과 같은 버킷 → 저장 → Retry deployment(재배포) 1회.

바인딩 전 = 기존 30MB 경로 그대로(아무것도 안 깨짐). 바인딩 후 = 뷰어가 자동 감지해 2GB 직업로드 켜짐(코드 추가 배포 불요). 선택 후속: R2 수명규칙(prefix up_src/ 1일 삭제) 추가하면 중단 잔재까지 자동 청소.
