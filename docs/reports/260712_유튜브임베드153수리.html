한 줄 결론. 트렌드 임베드가 4개 플랫폼(유튜브·틱톡·릴스·X)과 공유하는 iframe #tvFrame에 걸려 있던 referrerpolicy="no-referrer"가 유튜브 재생을 막았다. 유튜브 임베드 플레이어는 부모 origin(HTTP Referer 헤더)로 "어느 도메인이 임베드 중인지"를 검증하는데, no-referrer가 referrer를 통째로 지워 유튜브가 임베더를 식별하지 못하고(embedder.identity.missing.referrer) 오류 153("동영상 플레이어 구성 오류")을 던졌다. 브라우저 기본값 strict-origin-when-cross-origin으로 교체 = 경로·쿼리는 감추고 origin(https://apps.nomute.kr)만 전송 → 도메인 검증 충족 → 재생 복구. 계정 로그인/자격증명은 원래 불필요(임베드는 공개 재생)라 아무 자격증명도 저장하지 않았다.

4. 리스크 · 장단점

장점

  • 근본원인 정조준 — 153의 유일 원인(stripped referrer)만 겨냥, 유튜브 공식 권장값과 정확히 일치.
  • 최소 변경 — 1파일·1속성. 되돌리기 1초. 블라스트 반경 최소.
  • 회귀 0 — strict-origin-when-cross-origin은 브라우저 기본값 → 공유 iframe의 틱톡·릴스·X는 종전과 동일.
  • 자격증명 불필요 — 임베드는 공개 재생. 로그인·쿠키·토큰 저장 0 = 유출면 0.
  • 프라이버시 유지 — 경로·쿼리·토큰은 계속 감춤, origin만 전송.
  • 즉시 반영 — viewer/_headers no-cache라 하드리로드 없이 배포 즉시 라이브.

단점 · 한계(정직)

  • 153만 담당 — 소유자 임베드 금지(150/101)·연령제한·지역차단·삭제(100)는 referrer 무관 → 이 fix로 안 열림(유튜브 정책, 우리 버그 아님).
  • 임베드 실패 자동 감지 없음 — 금지 영상은 사용자가 눈으로 보고 헤더의 작은 '원문'을 스스로 찾아야 함(저발견성).
  • 폰 자동재생 미보장 — autoplay=1만으론 모바일 정책상 첫 탭 필요(음소거 자동재생 원하면 &mute=1 별도 — 153과 무관한 제품 결정).
  • trEmbed 정규식 협소 — watch?v=만 매치. youtu.be·shorts·live는 새 탭 폴백(별개 이슈).

리스크 & 완화

  • 서버 헤더 오버라이드(이론) — 미래에 Referrer-Policy: no-referrer 응답헤더를 깔면? → per-element referrerpolicy 속성이 그 iframe에 우선하므로 안전. (완화: 필요 시 <meta name="referrer"> 하드닝.)
  • null-origin 컨텍스트 — file://에서 열면 origin 못 보내 153 재발 가능 → 실운영은 HTTPS 배포·PWA라 해당없음.
  • 공유 iframe 부작용 — 4플랫폼 공유지만 기본값 복귀라 부작용 0.

새 아이데이션 (후속 · 이번 범위 밖)

  • 임베드 실패 자동 유도 — 유튜브 분기에만 IFrame Player API(enablejsapi=1+onError) 얹어 150/101 감지 시 본문을 "임베드 막힘 + 큼직한 유튜브에서 열기 CTA"로 스왑. (중규모·유튜브 전용.)
  • 정규식 broaden — youtu.be/·shorts/·live/·m.youtube도 trEmbed에서 임베드로. (트렌드 데이터 URL 형태 실측 후.)
  • 음소거 자동재생 — 운영자가 "열자마자 재생"을 원하면 유튜브 URL에 &mute=1(정책 허용). 제품 결정 사항.
  • meta referrer 하드닝 — head에 <meta name="referrer" content="strict-origin-when-cross-origin"> = 미래 서버헤더 방어 belt-and-suspenders(현재 불요).
