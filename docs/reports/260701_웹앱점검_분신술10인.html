nomute-editor 뷰어 · 기능 동작 + 성능('버퍼') 적대적 감사 · 2026-07-01 · READ-ONLY 실측

결론: 기능은 대체로 건강 · 최근 변경 회귀 0 · 하지만 '버퍼'는 실재하고 원인이 명확히 규명됨.

10개 앵글로 뷰어를 실측 감사. 알림·푸시 정합, 큐 동기화, 입력·모달 방어, 백엔드 인젝션/경로 방어는 이 레포에서 가장 잘 만든 축(대부분 ✅). 최근 4개 변경(정치 강마커·카드 진행문구·대기열 동기화·글래스)은 회귀·미배선 0, 뷰어 JS 구문 RC=0. 운영자가 지목한 '버퍼'(랙)는 진짜 있고, 4개 독립 에이전트 + 메인 실측이 같은 3가지 원인으로 수렴했다.

1. '버퍼' 진단 — 원인 3면 (교차검증·핵심)

뷰어가 무거워지는 인과: 리스트가 커질수록 매 폴(60초)·타이핑마다 전체를 다시 계산·다시 그리는데, 그걸 접는 장치(메모·디바운스·가상화)가 없다. 세 층이 곱해진다.

② 폴링·네트워크 계층 (JS) — 무조율 다중 폴러 중

  • pollPicking 백그라운드 좀비 index.html:5719-5738 — while 최대 11분, 탭 숨기면 continue로 fetch만 건너뛰고 루프는 안 죽음. 픽 완료해도 조기종료 불능 → 탭 숨긴 채 픽=11분 좀비.
  • load()·loader 인플라이트 가드 부재 2188 · loadPending·loadMessages·loadHealth·loadPickFails — livePoll(60s)이 6개 fetch 병렬, in-flight 가드는 loadCandidates 하나뿐. pollPicking(25s)·qFlash·visibilitychange가 겹치면 같은 요청이 중첩 발사.
  • pending.js 팬아웃 functions/api/pending.js — 요청당 GitHub API 최대 ~110호출. 동접 늘면 5000/h rate limit 소진 → 대기열이 조용히 빈 배열 폴백.
  • GitHub fetch 타임아웃·재시도 없음 발사 엔드포인트 전량 — spellcheck만 타임아웃 보유. 느리면 무응답(발사 실패도 못 냄).

③ CSS 컴포지트 계층 — 스크롤 페인트 중

측정: backdrop-filter 81 · box-shadow 121 · 무한 애니 22 · content-visibility 0 · 리스너 195 · setTimeout 112.

  • 피드 카드 비가상화 × N backdrop-filter .card :321 — 전 카드 DOM, 카드마다 blur+3겹 shadow 동시 라이브. content-visibility:auto 미적용(코드 최소·효과 최대인 최적화 빠짐).
  • 하단 네비 26px 블러 고정바 .bnav :780 — 레포 최대 블러+saturate, position:fixed라 스크롤마다 재샘플. reduced-motion으로도 안 꺼짐. 모바일 jank 1순위.

3. 조치 목록 (메인이 검증 거쳐 수정)

Tier 1 — 버퍼 직격 · 저위험 · 즉시 (권장)

  • .card {content-visibility:auto; contain-intrinsic-size} — 뷰포트 밖 카드 backdrop+shadow 페인트 스킵. 스크롤 jank 최대 ROI, 코드 최소.
  • .bnav 블러 26→14px + saturate 제거 검토 — 가우시안 비용 반토막, 시각차 미미.
  • .bannerframe will-change:filter 제거 + filter 애니 완화 — idle GPU 레이어 예약 해제.
  • 정렬키 Schwartzian 사전계산 — arr.map(c=>({k:scScore(c),c})).sort(...) = 카드당 1회. scTs·articleCat 메모(c._ts/_cat ??=).
  • 렌더 조기반환 — 시그니처 동일이면 DOM 재구축 skip(상태 클래스만 토글).
  • 자동완성 통일 — #askText/#revText/#crevText/#pasteText에 autocorrect/autocapitalize/spellcheck off.
  • 죽은코드 회수 — saveAll·closeToolFully(+문서 괴리)·edLaunchScene 등 5개.

Tier 2 — 폴링·네트워크 · 중위험 · 검증 필요

  • pollPicking 백그라운드 break(좀비 11분 제거) + 검색 150ms 디바운스.
  • loader 인플라이트 디둡 — livePoll·pollPicking·qFlash 중복 폴 병합(api/pending 인플라이트 공유).
  • 공통 ghFetch 유틸 — AbortController 타임아웃 + 429/5xx 백오프(발사 엔드포인트 전량).

Tier 3 — 아키텍처·큰 작업 · 운영자 결정

  • index.html 인라인 CSS/JS 분리 — 무거운 스타일·유틸을 ?v= 장기캐시 파일로. HTML 셸만 얇게(재방문 재검증 795KB→소). 최대 로드 ROI지만 큰 리팩터.
  • pending.js 경량화(raw 지연로드·edge cache) + candidates.json 3.3MB slim.
  • 발사 엔드포인트 originOk 가드 — ⚠️ 운영자 "지출 직접 모니터링" 정책과 상충 가능 = 확인 후.
  • check_refs 게이트 추가 — 강마커 py↔js 정규식 미러 + dead-JS 함수 검출.
  • dl.js SSRF 강화(정수/hex IP 표기 차단) · submit.js 이미지 합계 상한.

4. 💡 아이데이션 베스트픽

1. 렌더 코얼레싱 레이어 하나가 버퍼의 절반을 접는다. 폴러 전부를 "dirty 플래그 set → rAF/디바운스로 1회 load"로 라우팅하면 ①중복 fetch ②통짜 재구축 ③make 60초 지연이 동시에 완화된다. 지금의 단일-루프 가드 구조와 궁합이 좋다.

2. content-visibility:auto + Schwartzian 정렬 두 줄이 스크롤/폴 랙을 리스트 크기와 무관하게 만든다. 코드 최소, 픽셀 회귀 0.

3. popstate·자동완성은 SSOT 헬퍼로. 지금은 6개 전역 리스너 + 입력칸마다 손으로 박은 속성 = 새 오버레이/입력칸 추가 시 드리프트. 헬퍼 하나로 통일하면 회귀 취약성 제거.
