한 줄. 알림 3종을 누르면 이제 그 건에 정확히 떨어진다 — 긴급은 요약이 됐으면 요약창, 안 됐으면 가장 메이저 기사 원문으로(웹앱 경유) · 요약 완료는 그 요약 · 작업 완료는 그 제작 결과(중상단).

핵심 결함이었던 긴급 알림은 그동안 그냥 루트(피드 첫 화면)로만 떨어졌다 → 무슨 속보였는지 직접 찾아야 했음. 이제 딥링크로 해당 건 직행.

긴급 알림 = 어떻게 “해당 건”을 찾나 (쉬운 설명)

속보가 뜨면 보통 요약은 1~2분 뒤에 완성된다. 그래서 누른 그 순간에 요약이 됐는지 안 됐는지로 갈린다(서버가 미리 정하지 않고 탭 시점에 판단):

🚨 긴급 알림 탭 → 웹앱 열림 → 요약 됐나? → YES · 요약창 / NO · 가장 메이저 기사 원문(새 탭)

메이저 = 직전 작업의 조선·동아(보수메이저) → 중진보 메이저 픽(breaking_pick). 통신사·군소가 아니라 풀텍스트 종합지로 직행. 푸시에 &bl=메이저url을 같이 실어 후보 조회가 실패(랙·만료)해도 메이저 원문 보장.

분신술 = 적대적 검증 10인 → 핵심 결함 잡아 수정 만장일치 ✅

구현 후 Opus 10인을 서로 다른 앵글로 병렬 소환(서비스워커·매칭·타이밍·재진입·페이로드·썸네일·회귀·파이프라인·모바일·총괄). 발견 → 수정 → 재검증 3인 전원 ✅까지 루프.

장단점

✅ 장점

  • 3종 알림 모두 탭 한 번에 해당 건 직행(특히 긴급 = 루트 탐색 제거).
  • 긴급 미요약 시 풀텍스트 메이저 원문(통신사·군소 회피 — 직전 작업과 한 몸).
  • 포그라운드/백그라운드 동작 일치(같은 goBreaking).
  • 추가 비용 0 — breaking_pick은 이미 있던 데이터, 푸시 url만 바뀜.

⚠️ 한계·주의

  • 작업 완료(썸네일)는 가장 최근 결과로 착지 — 동시에 여러 개 만들면 그중 최신으로(보통 1개씩 = 무해). 건별 정밀은 푸시에 job-id 필요(후속).
  • 긴급 미요약 새 탭이 드물게 팝업 차단되면 동일탭 폴백(이미 URL 정리돼 루프 없음).
  • iOS는 홈화면 추가(PWA)해야 푸시 수신(애플 제약·기존과 동일).
