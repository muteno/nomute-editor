서술

1. #pubdlg 키보드 가림 — 이미 파일이 알고 있는 결함의 잔존 (상)

증상: 폰에서 발행 모달을 열고 PIN 설정 fieldset(창 하부)에 포커스하면 OS 키보드가 올라오면서 입력칸이 가려진다. 재현 조건: 390×844 iOS Safari에서 #pubdlg 열기 → PIN 입력 탭. 세로 모드에선 2801의 margin:8vh auto auto; margin:8dvh auto auto(≤600px) 상단 앵커링이 1차 방어를 하지만, 원인 실측을 보면 기저 2798행이 max-height:88vh로 vh 단위다. 이것이 왜 결함인지는 이 파일 스스로 증언한다 — 2849행 #genidlg 주석: "dvh 통일(적대검증C: 8dvh margin과 88vh 혼용 = 키보드 시 어느 경로도 미축소 → dvh로 일치)". 즉 운영자 적대검증에서 이미 '88vh 혼용'이 키보드 미축소 버그로 판정돼 #genidlg는 88dvh로 고쳐졌는데(2849), 같은 계보의 형제 모달 #pubdlg는 2798에 88vh가 그대로 남았다. 더 나아가 #genidlg·#tooldlg에는 visualViewport 클램프(geniVV 4819·toolVV 4823)가 배선돼 있으나 #pubdlg에는 어떤 VV 클램프도 없다. 601px+ 폭(폰 가로모드 844px·폴더블 펼침·소형 태블릿)에서는 2801 앵커링마저 탈락해 margin:auto 중앙 정렬 + 키보드 = 입력칸 완전 가림이 성립한다. 사용자 영향: PIN은 발행 잠금 해제라는 데이터 접근 관문이라, 입력칸이 안 보이는 채로 키패드를 두드리는 오입력 사고 가능 = '입력/데이터 사고 가능' 기준으로 상. 개선안: 값 창작 없이 2849를 그대로 계승해 88vh→88dvh 치환하고, geniVV의 gdlg 게이트를 pubdlg까지 일반화(toolVV가 이미 "geniVV의 셸 일반화"라고 자기 기술한 4823 주석의 확장 축과 동일). 부작용 리스크: dvh 치환은 주소창 수축 시 모달이 미세하게 커지는 리플로우가 생기지만 #genidlg가 동일 값으로 이미 운영 중이라 검증된 경로다.

2. geniVV 반쪽 클램프 — toolVV 주석이 지목한 함정에 자신이 빠져 있음 (상)

증상: 이미지 생성 팝업 자유모드에서 문구 입력(#geniWish) 포커스 시, 폰 가로모드(844×390)나 601~640px 폭에서 팝업 하단(입력칸·생성 버튼)이 키보드 아래로 사라진다. 재현: 가로모드로 #genidlg 열기 → 자유모드 문구 탭. 원인 실측: geniVV(4819-4822)는 dlg.style.maxHeight = Math.round(window.visualViewport.height * 0.92)+'px'로 높이만 줄인다. 그런데 세로 위치는 기저 2849의 margin:auto(≥601px에서) — 이 margin:auto가 무엇을 기준으로 중앙 정렬하는지를 같은 파일 toolVV(4828)가 정확히 명시한다: "margin:auto는 layout viewport 중앙 = 키보드 위 아님". 그래서 toolVV는 marginTop = Math.round(vv.offsetTop + vv.height*0.04)+'px'로 보이는 영역 상단 기준 재배치를 하는데, geniVV엔 이 두 번째 반쪽이 없다. 가로모드 산수: 세로 390px, 키보드 약 210px → vv.height≈180px, maxHeight 클램프=166px. 그러나 margin:auto 중앙 = layout viewport 390px의 중앙(195px 부근)에 놓이므로 모달 하단은 키보드 상단(180px)을 넘어 잠긴다. 세로 모드(≤600px)는 2852의 8dvh 상단 앵커가 우연히 구제하지만, dvh는 iOS에서 키보드에 반응하지 않으므로 이것도 '위에서부터 67px' 고정일 뿐 vv.offsetTop 추적이 아니다. 추가 실측: toolVV는 톱레벨에서 resize와 scroll 둘 다 청취(11084 — "scroll = 키보드 뜬 채 vv.offsetTop 이동 추적")하는데 geniVV는 4753에서 resize만 청취한다. 키보드 뜬 채 사용자가 화면을 끌면 vv.offsetTop이 변해도 geniVV는 안 불린다. 사용자 영향: 자유모드 생성 진입 자체가 막히는 조작 불가 시나리오 = 상. 개선안: geniVV에 toolVV의 60px 괴리 게이트·marginTop 보정·scroll 청취 3종을 그대로 이식(동일 파일 정본 복사 = [4] 계승 원칙 그대로). 부작용 리스크: genidlg close 시 인라인 원복은 4755에서 maxHeight만 지우므로 marginTop 원복 라인 추가 필요(10832의 tooldlg 원복 선례 계승) — 누락하면 다음 오픈 시 stale 위치.

3. viewport-fit=cover 선언 뒤 좌우 인셋 미소비 (중)

§1. iOS 포커스 자동 줌 — 스튜디오군 전 입력이 대상 (no.1~5 · 상 클러스터)

증상: 아이폰 사파리에서 텍스트 입력을 탭하는 순간 페이지 전체가 확대되고, 키보드를 내려도 확대가 풀리지 않아 사용자가 손가락으로 축소해야 원상복귀된다. PC와 안드로이드 크롬은 무증상 — 전형적 iOS 전용 이질감이다.

재현 조건: iPhone(390×844) Safari에서 nb.html:187 URL 입력, track.html:232 URL 입력, song.html:229 직접 입력, ly.html:534 찾기 입력, k.html의 장면 textarea 아무거나 탭.

원인 실측: iOS Safari는 포커스된 편집 요소의 computed font-size가 16px 미만이면 자동으로 페이지를 줌한다. 이 레포 스튜디오군의 방어선은 세 겹 다 뚫려 있다. ① 전역 층 — 7파일 전원 body가 font:15px/1.6 …(song:28·track:28·nb:29·ly:27·k:25·sb:24·conv:27)이고, 입력 전역 규칙이 textarea, input[type=text] { … font:inherit; }(song:40·nb:36·track:36·ly:36·conv:34) = 모든 입력의 기본값이 15px. 15px는 줌 배율 16/15≈1.07배라 미세하지만, 잔류 줌 특성상 한 번 발동하면 풀릴 때까지 레이아웃이 밀린 상태다. ② 오버라이드 층 — 핵심 입력일수록 더 작게 내려 잡았다: song .selin input 12px(82), k .sg/.dialwrap textarea 13px(198·205), ly .lyfind input 13px(164), track .nin 14px(161). 12px는 1.33배 줌 = 조작 붕괴 체감이 가장 크다. ③ 뷰포트 층 — 7파일 메타(각 :5)는 width=device-width, initial-scale=1, viewport-fit=cover로 maximum-scale 무지정 = 줌 억제 장치도 없다(억제 자체가 접근성상 비권장이므로 이 층은 그대로 두는 게 맞다).

사용자 영향: song의 직접 입력은 selpop 팝업(:56)이 열린 채 발동하므로, 팝업+줌+키보드 3중 겹침으로 화면이 뒤엉킨다. k는 장면·대사 textarea가 제작 흐름의 본체라 매 문장 입력마다 줌·해제가 반복된다. 빈도×강도 모두 최상급이라 '상' 판정.

부작용 리스크: 12→16px는 33% 확대라 selpop(360px 캡)·soptrow 내 줄바꿈 지점이 이동한다. .selin은 flex-basis:100%(81)라 폭 여유가 있어 안전. k textarea는 min-height:54px 안에서 보이는 줄 수가 3→2.5줄로 줄 수 있어 min-height 동반 조정(+8px 내외)이 필요하다. smoke_geni.js 등 표면 스모크가 폰트 픽셀을 어서션하고 있다면 게이트 값 갱신 필수.

§10. 회귀 리스크 매트릭스 — 게이트로 잡을 것과 눈으로 잡을 것

①(16px 승격)의 회귀 반경이 가장 넓다. 15→16px는 +6.7%라 미미하지만 12→16px(.selin)·13→16px(k textarea)는 +23~33% — 예상 지점은 selpop 내부 줄바꿈 이동(.selin이 flex-basis:100%(song:81)라 폭은 안전, 높이만 +8px 내외), k 카드의 세로 리듬 변화(min-height 보상으로 상쇄), 장문 placeholder(nb:187 "유튜브 링크 — …")의 말줄임 시점 앞당김이다. [15] 상비 게이트 축으로 보면 smoke_*.js는 파일 헤더에 담당 표면을 선언하는 구조이므로 이 다섯 파일이 등재 표면인지 커밋 전 확인이 필요하고, 미등재라면 [4-1]의 "신설·변경 = baseline 즉시 등재" 축에 따라 "입력 computed font-size ≥16" 어서션을 추가할 기회다 — 기하·computedStyle 어서션만 허용하는 게이트 규약과 정확히 정합하는 유형이다. ②(tblwrap)는 DOM 구조 변경이라 셀렉터 의존 코드를 스캔해야 하는 부류지만, 표 안에 버튼·리스너가 없음은 이미 실측했다(복사 바인딩은 .code button 한정 — ly:808) = 반경 0에 수렴. ③(hpop absolute)은 스크롤 후 열기 시 팝업이 문서 상단에 남는 트레이드오프가 유일한 실질 리스크 — .histbtn 자체가 문서 상단 고정이라 체감 확률은 낮고, 우려되면 트리거 rect 기준 JS 배치로 승격한다(비용 중). ④(히트 확장)는 시각 무변경이라 픽셀 게이트 무풍이지만, .lyseg-t의 행 간 gap이 확장 inset보다 작으면 인접 행과 히트가 겹쳐 오탭이 역전된다 — 적용 전 행 gap 실측이 선행 조건. ⑤(env 합성)는 max() 폴백으로 비노치 기기 무변경이 보장되고 iframe 구동 시 env()=0이라 이중 여백도 없다 = 반경 최소. 종합: 값 승격(①)은 기계 게이트로, 좌표계 변경(③)은 눈 실측으로 — 검증 축이 갈린다.

§11. 우선순위 논증 — 왜 이 순서인가

빈도×강도×수정비용 3축으로 세운다. 1순위 = iOS 줌 클러스터(no.1~5): 발동 빈도가 "입력할 때마다"로 최상이고 — 파일 내 서열은 k(장면·대사 = 제작 본체) > song(옵션 4개소) > ly(찾기/바꾸기) > track·nb·conv(URL 1회성) — 강도는 잔류 줌이 후속 전 조작을 오염시키는 수준, 수정비용은 CSS 값 10곳으로 최소다. 비용 대비 회수 면적이 압도적이라 단독 선두. 2순위 = ly 표 오버플로(no.6): 빈도는 AI 산출 의존이라 확률적이지만 발동 시 페이지 전체 조작이 오염되고([15] 금지 축 직접 위반), 수정이 래퍼 2줄이라 역시 저비용이다. 3순위 = hpop 좌표계(no.7): PC 단독 진입 한정이라 빈도는 낮으나, 15건 중 유일하게 "기능이 없다"로 오인되는 항목(팝업이 시야 밖) = 기능 손실 오인 축이라 중 상단에 둔다. 4순위 = 히트 확장·safe-area(no.8~10): 상시 미세 마찰이라 신고로 올라오지 않는 유형 — [4-1]의 touch-then-align 원칙대로 해당 파일을 다른 일로 만질 때 곁들이는 소화가 정합적이다. 티어 불일치(no.11·12)는 동결 정본 원칙상 소급 캠페인 금지를 재확인하며 최후순위 고정. 이 순서의 함의는 "폰 사용자 보호는 전역 규칙 층에서 먼저, 좌표계·디테일은 표면별로 나중에"이며, 7파일이 body·input 전역 규칙을 동형으로 공유하는 구조가 1순위를 10곳 수정으로 끝내게 해주는 지렛대다.

총평

이 스튜디오군의 모바일 설계는 골격 자체가 모바일 퍼스트(600px 단열·유동 폭·터치 액션 방비)라 레이아웃 붕괴급 사고는 없다. 대신 결함이 한 겹 아래에 균일하게 깔려 있다 — 입력 폰트 16px 미달이 전역 규칙 층위에서 전 파일 공통(no.1~5)이라는 점이 이번 감사의 핵심이고, 이는 파일별 땜질이 아니라 전역 input 규칙 5곳의 값 승격으로 뿌리째 잡힌다. 나머지는 "규약은 있는데 절반만 적용된" 유형이 지배적이다: §16 히트 확장이 .spbtn엔 있고 .selv엔 없고, break-word가 .code엔 있고 table엔 없고, 260713 토큰 승격이 song엔 있고 nb엔 없다. 즉 신규 설계가 아니라 레포가 이미 가진 문법(ly:230 16px·song:79 히트 확장·k:130 페이드 레일·index:1919 세이프 대응)의 잔여 전파가 개선의 전부이며, [4-1]의 touch-then-align 원칙 안에서 소화 가능한 규모다. 우선순위는 iOS 줌 클러스터(상 3건) → ly 표 오버플로·hpop 좌표계(중) → 히트 확장(중) 순을 권고한다.

증상: iPhone(390×844, 홈바 기기)에서 edit의 "처리 추가" 바텀시트를 열면 목록 마지막 행이 홈 인디케이터와 겹친다. 재현: 폰 세로, 시트 오픈, 끝까지 스크롤 — 마지막 행 탭이 홈 제스처와 경합한다. 원인 실측: edit.html:116 .sheet { position:fixed; left:0; right:0; bottom:0; height:64dvh; … padding:14px 16px; overflow-y:auto; } — 하단 패딩이 고정 14px(축약 padding의 상하값)이고 env() 가산이 없다. edit.html:5가 viewport-fit=cover라 시트 박스 자체가 홈바 뒤까지 내려간 상태다. PC에서는 홈바가 없어 완전 정상 = 전형적 "PC에서 안 보이는 폰 사고". 사용자 영향: 오탭 시 앱 전환 제스처 발동 = 작업 흐름 끊김, 최악은 마지막 항목 선택 불가 체감. 개선안: index.html:1476 하단 네비의 확립 패턴 padding:0 8px max(6px, env(safe-area-inset-bottom)) — "가산이 아니라 흡수(max)" 주석까지 달린 정본 — 를 시트 하단에 계승한다(padding:14px 16px max(14px, calc(8px + env(safe-area-inset-bottom))) 류). 부작용 리스크: 홈바 없는 안드로이드에선 env()=0이라 max()가 기존값 유지 = 회귀 0. 단 시트 height:64dvh는 "운영자 선택값" 주석이 있으므로 높이는 불변, 패딩만 만진다.

2. 640/600 이중 경계 — 같은 화면 안 반쪽 반응 (no.2 · 중)

증상: 폭 601~640px 뷰포트(갤럭시 폴드 펼침 632px, iPad 분할뷰, PC 좁은 창)에서 index의 어떤 다이얼로그(1697·1836·2801·2852 계열, 예: 1836 @media (max-width:600px) { #editdlg { width:94vw; } })는 PC 규격으로, 어떤 다이얼로그(868·916·951·1039·1059·1919·2511 계열)는 폰 규격으로 렌더된다. 같은 앱에서 팝업마다 폰/PC 판정이 갈리는 상태 — 요구된 "이질감의 구조 원인" 그 자체다. 원인: 경계값이 :root 토큰처럼 정본화되지 않아 작성 시점마다 640 또는 600이 임의 선택됐다(2925의 480은 초소형폰 전용 3차 분기라 별개 축). index.html:1552 주석이 이 병리의 실사고 전례를 자백한다: "분기하면 한쪽만 조정돼 재발했다"(스크림 뷰포트 분기 폐지 사유). 사용자 영향: 폴더블·태블릿 사용자에게 모달마다 크기 문법이 다른 앱으로 보임. 개선안: 경계 1값(640 권장 — 7:4로 다수파)을 :root 인접 주석 정본으로 선언하고, 600 분기 4곳은 [4-1] touch-then-align 원칙대로 해당 파일을 다른 일로 만질 때 개별 실측 후 흡수(일괄 캠페인 금지 — CSS는 var()를 @media 조건에 못 쓰므로 기계 강제 불가, check_refs에 텍스트 스캔 축으로 등재 가능). 부작용: 600→640 승격 시 601~640 구간이 폰 모드로 바뀌는 게 의도된 개선이지만, PC 좁은 창 사용자에게는 다이얼로그가 갑자기 94vw로 커지는 변화 = 커밋 분리 필수.

3. 플레인 vh 잔존 6곳 — dvh 규율의 미전파 (no.3 · 중)

증상: 폰 Safari/크롬에서 주소창이 펼쳐진 직후(페이지 진입 직후가 대부분) 고정 팝업의 하단이 화면 밖으로 잘린다. 재현: song 히스토리 팝업(.hpop, song.html:165 max-height:min(70vh,480px)) 진입 직후 오픈 — 70vh가 대형 뷰포트 기준이라 실가시 높이보다 ~9% 크다. 동형: nb.html:162(70vh), conv.html:100(62vh)·120(56vh), track.html:183·184(60vh)·193(62vh). 이 레포는 dvh 규율이 명시적으로 확립돼 있다 — thumb.html:734 주석 "84vh→84dvh 주소창 대응(분신술10 260704)" — 그런데 전파가 대형 파일(index·thumb·edit·ly)에서 멈추고 소형 스튜디오 4파일에 구세대 vh가 잔존한 것. 흥미로운 실측: track.html:59는 40vh와 40svh를 한 줄에 병기(폴백 패턴 = dvh/svh 미지원 구형 브라우저 안전망)하는 모범인데, 같은 파일 183·193은 플레인 vh다 — 파일 내부조차 혼재. 사용자 영향: 팝업 하단 버튼·마지막 행이 주소창 뒤에 숨어 "스크롤해도 끝이 안 나오는" 체감. 개선안: track:59 병기 패턴을 정본으로 6곳 치환(max-height:min(70vh,480px); max-height:min(70svh,480px); 2줄). 부작용 리스크: svh는 주소창 축소 후에도 소형 기준 고정이라 팝업이 다소 짧아짐 — 고정 팝업 max-height 용도로는 안전한 방향(잘림보다 여백이 낫다).

4. min-aspect-ratio = PC 프록시 오발 (no.4 · 중)

증상: 폰을 가로로 돌리면(844×390, aspect 2.16:1) index 이미지 편집 다이얼로그가 "PC용" 규격에 걸린다. 원인: index.html:1837 @media (min-aspect-ratio:1/1) { #editdlg { max-height:66vh; } .ed-img { max-height:24vh; } .ed-body { max-height:calc(66vh - 114px); } } — 주석 "PC(가로>세로) = 창 ~60%대". 가로>세로를 PC 판정으로 쓰는 산식인데 폰 랜드스케이프도 참이다. 390px 높이에서 .ed-img 24vh = 94px 썸네일, .ed-body = calc(257px-114px)=143px — 조작은 되지만 이미지 확인이라는 목적이 무력화된다. 겹침 사고: 이 분기만 vh(dvh 아님)라 가로 폰의 주소창 변동까지 얹힌다. 사용자 영향: 이미지를 크게 보려고 가로로 돌렸는데 오히려 더 작아지는 역설. 개선안: 조건을 (min-aspect-ratio:1/1) and (min-width:640px)로 조이면 폰 랜드스케이프(폭 844)는… 여전히 매치한다 — 정확히는 (min-width:640px) and (min-height:500px) 같은 높이 가드가 본질(폰 가로의 본질은 "높이 부족"). 동시에 66vh→66dvh. 부작용: PC 세로 모니터(9:16 회전 모니터)가 폰 규격을 받게 되는 엣지 — 드문 케이스라 수용 가능.

6. 폰 로딩 부담 — 1.6MB 단일 HTML + 인라인 폰트 2본 (no.6 · 중)

index.html 1,639,930B(ls 실측), thumb 442,943B. index에는 woff2 base64가 2본(123 본문용, 11547 SUMMARY_TPL 주입용 — %FONTFACE% 치환 소스) 인라인돼 있고 base64 총 8개소다. gzip 후에도 폰 LTE 첫 페인트가 PC 대비 수 배 늦고, 1.6MB HTML 파싱은 구형 안드로이드에서 메인스레드를 수백 ms 점유한다. base64는 gzip 효율도 나쁘다(+33% 인코딩 팽창). 발행물용 두 번째 폰트는 본문 렌더에 불필요한 사본이므로 발행 시점 fetch 주입으로 빼는 것만으로도 즉효. 부작용: 단일 파일 배포 단순성(정적 뷰어 계약)이 이 레포의 의도적 아키텍처라, 외부화는 오프라인 열람·파일 복사 시나리오를 깨는지 운영자 확인이 선행돼야 한다 — 사본 제거(SUMMARY_TPL 쪽)가 무갈등 1수다.

7. SUMMARY_TPL 100vh — 발행물만 구세대 (no.7 · 중)

레포 전체에서 100vh는 단 1곳, index.html:11392 .nm-root { position:relative; min-height:100vh; padding:30px 16px 56px; overflow:hidden; } — 발행 요약 페이지 템플릿이다. 본체가 dvh 전면 규율(862·916·1059·2492 등 수십 곳)인 것과 대조된다. min-height라 콘텐츠가 길면 자라므로 잘림 사고는 없지만, 콘텐츠가 짧은 요약에서 iOS 주소창 펼침 시 첫 화면 하단 배경 경계가 어긋난다. 발행물은 외부 공유 링크로 폰에서 열리는 게 주 소비 경로라 본체보다 오히려 폰 노출이 크다. 개선: min-height:100vh; min-height:100dvh; 2줄 폴백(발행물 = 수신자 브라우저 통제 불가 = 폴백 필수). 부작용 0에 수렴.
