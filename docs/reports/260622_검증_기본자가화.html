∴ 이제 적대적 평의회 소환은 명시·기틀·실질 코드/구조 변경(10인)에만 남음. 나머지(일상 수정·국소 뷰어·순수 룰·문구)는 자가검증.

장점 · 효율성

  • 오버헤드↓ — 일상 수정·국소 뷰어 손질마다 5명 에이전트 병렬 소환하던 비용·지연 제거(평의회 1회 = Opus 5~10 동시 발사).
  • 운영자 피로↓ — 작은 변경에도 평의회 평결·만장일치 루프를 기다리던 마찰 제거. 머지가 빨라짐.
  • 원래 효용 작던 영역 — §✅ 한계에 명시돼 있듯 순수 룰·문구·국소 손질은 에이전트 검증 효용이 작음. 자가검증(기계 게이트 + read-back)이 더 들어맞음.

품질저하 리스크 · 완화

  • 평의회 안전망 축소 — 국소 뷰어 손질의 잠재 회귀(다운스트림·모바일·메모리 사각)를 사람·에이전트 다앵글로 잡던 그물이 빠짐.
  • 완화① = 기계 게이트 유지 — check_refs(경로·버전·디자인거울·viewer JS 구문 하드게이트) + PR마다 도는 check-refs.yml CI는 그대로. 구문 깨짐·드리프트는 여전히 차단.
  • 완화② = 기틀급은 여전히 10인 — :root 토큰·배너 크로스페이드·파이프라인을 손대면 ⓒ에서 제외 = 10인 평의회. 동작·로직·상태가 한 줄이라도 끼면 면제 아님.
  • 완화③ = read-back 실측 불변 — 머지 후 git show origin/main로 원격 실내용 직접 확인하는 §수정모드 ⑤는 유지.
