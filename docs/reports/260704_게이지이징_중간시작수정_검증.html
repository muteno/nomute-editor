운영자 신고 = ① 게이지바가 중간에서 시작해 제대로 안 됨 · ② 아까 고른 센 이징이 아예 구현 안 됨.

진단 결과 = 배선·토큰 resolve 버그 아님(토큰은 :root→tokens.css→base.css→4뷰어 정상 전파, 애니메이션도 정상 발화). 곡선 값 자체가 계단함수에 가까웠던 것이 원인 — 옛 cubic-bezier(.12,.9,.22,1)은 시간 10%에 이미 55% 채움. 0.72초 게이지에서 첫 3~4프레임(~72ms) 만에 화면 중반을 지나 순간 텔레포트 → 눈엔 "중반부터 시작·이징 없음"으로 보임. 두 증상이 같은 뿌리. → 하우스 곡선 cubic-bezier(.22,.61,.36,1)(cardIn·bnav 계승)로 교체해 처음부터 눈에 보이게 감속하면서도 응답시점 71% 프론트로드 유지.

R-09 · 기존 → 현재 (전/후)

기존 (오전 PR #1536)

--gauge-ease: .12,.9,.22,1
  • 시간 10% → 55%, 25% → 83% 채움
  • 0.72s 게이지: 첫 4프레임에 중반 통과 → 순간 완료처럼
  • 눈에: "중간에서 시작 · 이징 없음"

현재 (본 수정)

--gauge-ease: .22,.61,.36,1
  • 시간 10% → 27%, 25% → 59%, 33% → 71%
  • 0부터 좌→우 보이며 감속(가시 채움)
  • 응답시점 71% 프론트로드 = 옛 linear(33%)보다 훨씬 채워짐 → 점프도 완화

한 토큰(:root --gauge-ease)만 교체 → build로 tokens.css·base.css·4뷰어(thumb/k/ly/comp) 전 게이지바(전송·수정·발행·기록·재생성·PIN·PICK·슛)에 일괄 전파. 제각각 없음(SSOT).

R-10 · 리스크

  • 주관적 강도 — "센"의 체감은 취향. 새 곡선은 강-중 감속이라, 옛 극단만큼 "확 참"을 원하면 약하게 느낄 수 있음(→ 보완 §R-12).
  • glive 응답 게이지 — 3.6s 점근 게이지도 같은 토큰 사용. 71% 프론트로드로 문제 없으나 실기기(모바일 GPU)에서 실제 fetch 타이밍 조합은 눈으로 재확인 권장.
  • reduced-motion — OS "동작 줄이기" 켜져 있으면 게이지가 .2s로 축약돼 여전히 순간처럼 보임(곡선 무관). 만약 운영자 기기가 이 설정이면 곡선 교체와 별개 요인.
  • 회귀 범위 — 토큰 1개라 전 게이지 동시 변경. 의도지만 넓음.

R-11 · 장단점

장점

  • 두 증상(중간시작·무이징)을 한 값으로 동시 해결
  • 새 곡선 창작 아님 = 기존 하우스 곡선 계승(§🎨 드리프트 0)
  • 옛 linear 문제(응답 점프)도 함께 완화
  • 토큰·과금 0(순수 CSS)

단점

  • 운영자가 오전에 고른 곡선을 덮어씀(재선택 필요할 수 있음)
  • 단일 토큰이라 게이지 유형별 미세 최적화는 불가(고정형 0.72s ↔ 점근형 3.6s 타협값)

R-12 · 단점 보완 방안

  • 강도 재튜닝 1줄 — 더 "센" 걸 원하면 .16,.8,.28,1(33%→84%)로 즉시 교체 가능. 위 데모에 곡선만 바꿔 재확인.
  • 유형 분리 옵션 — 정 필요하면 --gauge-ease(고정 0.72s·가시성)와 --gauge-ease-approach(점근 3.6s·프론트로드) 2토큰 분리(갱신=운영자 승인+거울+락). 지금은 과설계라 보류, 단일 유지.
  • 재선택 데모 상시화 — 본 보고서의 라이브 3열 비교를 곡선 튜닝 표준 도구로 재사용(재설계 0).

R-14 · 아이데이션

  • 곡선 미리보기를 뷰어 개발 토글로 — 숨은 debug 패널에서 --gauge-ease를 실시간 슬라이더로 바꿔 라이브 게이지에 즉시 반영(택1 후 값 복사). 취향 조율을 데모↔코드 왕복 없이.
  • 게이지 "채움 완료" 마이크로 피드백 — 100% 도달 순간 아주 짧은 밝기 펄스(0.12s)로 "다 찼다"를 명확히(끝이 완만해 끝점이 흐릿한 이징의 약점 보완).
  • 응답 도착 시 잔여구간 가속(이미 gdone .3s linear로 존재) — 곡선과 물리는 지점을 문서 다이어그램화해 다음 세션 혼동 차단.

R-15 · 신의 한수

증상을 "곡선이 틀렸다"가 아니라 수치(시간%→채움%)로 계량해서 "10%에 55%면 인간 눈엔 텔레포트"라는 물증을 잡은 것. 덕분에 새 색·새 곡선을 창작하지 않고 이미 코드에 있던 하우스 곡선(.22,.61,.36,1)이 정확히 이 문제의 해임을 %표로 증명 → §🎨 계승 원칙과 완전 정합하면서 두 증상·옛 linear 문제까지 한 값으로 봉합. "무이징처럼 보임"의 정체가 배선이 아니라 지각(perception)이라는 걸 분리해낸 게 핵심.

R-16 · 이건 모르겠다

  • 운영자 기기의 reduced-motion 설정 여부 — 켜져 있으면 곡선과 무관하게 게이지가 여전히 순간처럼 보임. 이게 원인의 일부인지 확실치 않음(코드로 확인 불가·운영자 기기 설정).
  • "센"의 정확한 목표 강도 — .22,.61,.36,1(강-중)이 딱 맞는지, .16,.8,.28,1(더 센)이 취향인지는 운영자 눈이라야 확정. 수치상 둘 다 "보이는 이징"은 충족.
  • 오전 데모에서 실제로 "B"가 어떤 곡선이었는지 — 시안 파일이 세션 스크래치라 소실. 선택된 B가 점근(3.6s)게이지 기준으로 보였고 고정(0.72s)게이지엔 부적합했을 가능성이 큼(추정).
