쉬운 설명 — 뭘 한 건가

트렌드 수집기가 원래 유튜브·구글트렌드·틱톡 + 구독 4축(X·틱톡·인스타·유튜브)을 긁었는데, 여기에 레딧 핫글·블루스카이 인기글·스레드 구독 계정 3개 축을 추가했다. 핵심 판정 하나: 스레드는 인스타랑 같은 Meta라 GitHub 서버(데이터센터 IP)를 차단한다 → 그래서 스레드만은 서버에서 안 긁고, 지금 X·인스타를 폰에서 긁어 밀어넣는 그 경로(phone_subs)에 태웠다. 맥에서도 같은 스크립트가 그대로 돈다(crontab 한 줄 · 크롬 로그인 불요 — 게스트 HTML 파싱). 레딧·블루스카이는 서버에서 무키로 바로 되는 것들이라 러너에 직접 붙였고, 셋 다 기본 꺼진 상태(게이트 OFF)로 머지해서 라이브는 아무것도 안 바뀌었다 — 켜는 건 카나리아 실측 후.

전 / 후 비교

전 (260712 오전)
  • 수집: 유튜브·구글트렌드(KR+월드)·틱톡·쇼츠·AI영상 + 구독 4축
  • 텍스트 SNS = X 하나(러너 429 로터리·폰 보조)
  • 해외 커뮤니티 반응 축 = 없음
  • 스레드·레딧·블루스카이 = 코드 흔적 0
후 (PR #2065)
  • + reddit[] 서브레딧 핫(popular·korea·worldnews 기본 · env 교체 가능)
  • + bsky[] What's Hot 인기(무키 · 데이터센터 IP 친화)
  • + subs.threads[] 구독 5축(폰/맥 공급 · 계정 파일 스키마·API 선대칭)
  • 산출 규격 = 기존과 동일(델타·순위이력 rh·first_seen 도장)

스레드: 폰/맥 crontab(30분) → 게스트 HTML walk 파싱 → sns_subs_phone.json → main 푸시 → 러너가 신선(90분)하면 채택 → sns_trends.json subs.threads

장 / 단점

장점
  • 전부 무키·무과금·LLM 0콜(기존 불변 유지)
  • 스레드의 Meta 벽을 새 인프라 없이 기존 폰 경로 재사용으로 통과
  • 스레드 파싱 = doc_id 하드코딩 대신 재귀 walk → 레이아웃 변경 내성

단점 · 리스크
  • 레딧이 러너 IP를 403/429할 가능성 — 카나리아가 판정(그래서 OFF 시작)
  • 스레드 = 비공식 게스트 파싱 = Meta 정책 변경에 취약(fail-soft라 죽어도 [] + 경고)
  • 블루스카이 한국어 밀도 낮음(글로벌 위주) — 가치 판단은 실데이터 보고
  • 컨테이너 프록시가 SNS 도메인 차단이라 이 세션에선 라이브 엔드포인트 실측 불가 → Actions 카나리아가 실측 정본

효율성 · 품질저하 리스크

  • 비용: 토큰 0 · API 키 0 · 과금 0. 러너 소요 +약 10~15s(레딧 3콜×2s + 블스 1콜) = 기존 timeout 840s 여유 내.
  • 품질저하 0 설계: 게이트 OFF = 함수 미호출(완전 무접촉) · 큐레이션 신호·임계·랭킹 0 접촉(표시 전용 레인 불변) · 기존 산출 키 바이트 무변경 · 뷰어 렌더 무접촉.
  • 검증: 몽키패치 스모크 16/16 + main() 통합 3케이스(전실패 no-op 보존 · 게이트 ON 키 정합 · 실패 런 직전분 보존) + check_refs 전 게이트 rc=0 + 머지 후 origin/main read-back.
  • UI/UX 변경 = 없음(뷰어 무접촉) → 디자인 전후 이미지 제출 대상 아님 · 신규 레인 화면 등장은 §디자인 j 배치 승인 후 별건.
