3. 가장 큰 "낭비"의 진실 — 이미 절반은 자동으로 해결돼 있었다

매 호출이 거대한 지침을 같이 보냄(요약 ~3.7만 토큰 / 카드 ~4.2만 토큰). 네 직감("같은 지침 두 번 읽힘")이 정확했어. 그런데 — 평의회가 claude CLI를 직접 돌려보니:

1번째 기사→지침 풀값 청구 (cache_w≈56K) 그 다음 5분 내 2·3·…번째→캐시로 ~1/10 값 (cache_r)

즉 한 잡에서 기사를 연달아 처리하면 2번째부터는 지침이 이미 싸게 청구되고 있을 가능성이 큼(서버측 자동 캐시·5분 유지). "두 번 읽힘"이 실제로는 "한 번 풀값 + 나머지 캐시"일 수 있다는 뜻.

내가 처음 걱정한 그림

기사마다 ~7.9만 토큰 지침을 매번 풀값 재청구 = 큰 낭비

실측이 말하는 그림

첫 기사만 풀값, 나머지는 캐시(~1/10). 실제 낭비는 측정해 봐야 안다 → 계측이 cache_r로 보여줌

⚠️ 솔직히 정정: 처음 "매번 풀 재전송"이라던 내 진단은 과했어. 캐시가 자동으로 걸려서 상당 부분 이미 절약 중일 수 있음. 그래서 계측부터 한 거고, 1~2번 돌면 cache_r 숫자로 진짜 낭비 규모가 나와. 그때 R6(아래)로 캐시를 "안 깨지게" 지키는 게 정답.

4. 안전한 절감 딱 하나 — R6 (승인 요청)

문제: 지침/룹브릭이 의미는 그대로인데 겉모양만(공백 추가, 파일 이름 변경 등) 바뀌어도, 시스템이 "지침이 바뀌었다"고 보고 같은 기사·카드를 전부 다시 만든다.

실측 전례: 파일명 rename 한 번에 카드 38장이 재촬영됐음(CLAUDE.md에 기록). 토큰·시간 낭비 + 이건 캐시도 깨뜨림(R4 절약까지 날림).

전 (현재)

지침 파일에 공백 한 칸 추가 → 버전 해시 바뀜 → 전 기사·카드 재생성 + 캐시 무효화

후 (R6)

의미 내용만 해시(공백·파일경로 제외) → 겉모양 변경엔 재생성 안 함 → 진짜 내용 바뀔 때만 재생성(드리프트 방어는 유지)
항목내용
효율성가끔 터지던 대량 재생성 스파이크 제거(rename·공백 편집 시) + R4 캐시 적중률 보호
품질 저하 리스크낮음·통제 가능 — 보수적 정규화만(앞뒤 공백·연속 빈줄만 정리, 들여쓰기 등 의도된 포맷은 보존). 진짜 내용 변경 시 재생성은 그대로 → 드리프트 방어 유지
왜 승인 필요inject_guidelines.sh(지침 주입 SSOT)를 건드림 = 기틀 인접. 규칙상 네 확인 + 변경 시 분신술 10인 재검증 필요

🔔 결정 필요 (R6)

이거 하나만 품질 0으로 안전하게 절감 가능해. 진행할까? 진행하면 별도 PR로 보수적 정규화 + 분신술 10인 재검증 후 머지. 안 해도 손해는 "가끔 겉모양 변경 시 재생성 스파이크" 정도(평소엔 무해).

🛡 품질 저하 0 — 어떻게 지켰나

남긴 건 품질과 무관한 것뿐: ① 계측(측정만·머지됨) ② R6(겉모양 변경 재생성 방지·승인 시) ③ R4(이미 자동 캐싱·측정으로 확인). 채점 effort max·2단계 구조·원문 의존·중복 방지 보수성은 전부 그대로 둠.
