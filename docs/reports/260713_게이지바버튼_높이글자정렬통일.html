쉬운 설명 (뭘 했나)

사용자가 기준으로 삼은 Image Studio [생성] 버튼(thumb .go)에 맞춰, 전 리포지토리 전폭 발사 게이지바 버튼 8개를 통일했다.

① 높이 통일 — 전부 52px로 맞춤(padding:14→18px).
② 글자 정중앙 — line-height:1을 넣어 글자의 실제 보이는 중심(ink)이 버튼 정중앙에 오게 했다. 예전엔 글자가 라인박스(폰트 여백) 때문에 살짝 위로 떠 있었다(−0.2~−0.5px).
③ conv·nb 3중 교정 — 이 둘만 높이(pad13)·모양(각진 r-s 9px)·글자크기(15px)가 정본과 달라서 통일(pad18·알약 pill·14px).

게이지 채우는 동작(goFill/glive)과 생성 → ✓체크 → 전송됨 → 원복 상태 흐름은 1바이트도 안 건드렸다(외형·정렬만).

선 그어서 검증 (헤드리스 실측 · 사용자 요구)

측정: 헤드리스 Chromium + canvas ink-metric으로 「글자 시각중심」과 「버튼 중심」을 계산해 빨강 파선=버튼 4분할 중심 / 청록·노랑 실선=글자 4분할 중심을 실제로 그어 비교.
BEFORE — thumb dy −0.2 · conv/nb dy −0.5 (글자가 위로 뜸)
AFTER — 8뷰어 전부 dy 0·dx 0·h 52 (십자선 정중앙 교차)

장점 · 단점 · 리스크

장점
  • 전 뷰어 발사 버튼 높이·글자 위치가 완전 일관(52px·dy 0) — 앱 전환 시 버튼이 미세하게 달라 보이던 드리프트 제거.
  • conv/nb의 실제 어긋남(각진 모서리·15px 글자·낮은 높이) 교정 = 명백한 버그 수정.
  • 정렬 기준이 코드 주석 ⭐정본 + 실측 게이트로 SSOT화 → 다음 세션이 또 어긋내기 어려움(§디자인 y 텍스트판).
단점 · 트레이드오프
  • 버튼 세로 여백(패딩)이 14→18px로 커짐. 높이 결과는 동일(52px)하나, 글자 위아래 빈 공간이 조금 늘어난 셈(line-height를 줄인 만큼 padding으로 보충).
  • index의 인라인 소형 게이지(.asksend 전송·발행·기록 등)는 이번 통일 대상이 아님 — 컨텍스트(모달 안 작은 버튼)가 달라 52px 전폭으로 못 키움. 단 이미 grid+lh1로 글자 정렬은 돼 있음.
품질저하 리스크 (점검함)
  • 게이지 채움·상태머신 로직 무접촉 → 동작 회귀 없음(CSS 정적 속성만 변경).
  • .go .gck(완료 체크)·.checking은 자체 grid place-items:center override라 lh 변경 영향 없음. ly .go svg(접수 체크 20px)는 순간 표시라 영향 미미(후속 미세조정 가능).
  • 다자식(텍스트+시간 gtime 등) 안전 위해 display는 block 유지(grid로 안 바꿈) — 세로 쌓임 리스크 원천 차단.

효율성

순수 CSS 3속성(padding·line-height, conv/nb는 +radius·font) 변경 = 런타임 비용 0·번들 증가 0. 검증은 헤드리스 1회(playwright-core)로 8뷰어 동시 실측. LLM·파이프라인·과금 무관.
