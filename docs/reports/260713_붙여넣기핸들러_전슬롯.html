1. 진단 — 무엇이 누락이었나

클립보드 이미지 paste 핸들러가 요약요청(#askText) 한 곳에만 있었다. 전수 조사 결과 뷰어 이미지 첨부 슬롯 6곳 중 5곳(geni 참고·편집 첨부·포스트 배경·자막·카드뉴스 배경)에 붙여넣기 핸들러가 없어, 파일 선택으로만 이미지를 넣을 수 있었다. 운영자 지적 정확.

영상·오디오 파일 슬롯(conv·edit·track·ly·song)은 클립보드에 그 바이너리가 담기지 않아 이미지 paste 대상이 아님 → 스코프 제외(이미지 첨부 슬롯 전수만).

3. 방식 — askText 선례를 공통 헬퍼로

  • 공통 헬퍼(각 파일 1쌍): _imgFromClipboard(e) = paste에서 첫 이미지 File 추출 · bindImgPaste(el, onImg) = el에서 Ctrl+V 이미지면 onImg(file) 호출. 이미지 아닌 paste·텍스트는 통과(preventDefault는 이미지일 때만) · 이중 바인딩 가드(_imgPasteBound).
  • 짝 입력칸이 있는 슬롯(geni·포스트·자막): 그 textarea에 배선 = #askText 선례 그대로. bindSlotImg는 read(f) 경로를 추출해 파일선택·붙여넣기 공용화 → 포스트·자막 두 슬롯 동시 커버.
  • 짝 입력칸이 없는 슬롯(편집·카드뉴스 배경): 편집은 모달(#editdlg) 레벨 — 모달 안 어디서 Ctrl+V해도 이미지면 첨부(edWish가 disabled 돼도 커버) · 카드뉴스 배경은 label이라 포커스 불가 → app5 노출(#imgBox:not(.none))일 때만 document 위임(타 탭 슬롯과 충돌 0).

4. 장점 / 리스크

장점

  • 일관 입력 — 이제 모든 이미지 슬롯에서 스크린샷 Ctrl+V 한 번으로 첨부(파일 저장→선택 왕복 제거).
  • 단일 헬퍼 — 슬롯마다 제각각 로직 창작 0, askText 선례 공통화 = 드리프트 축 없음.
  • 안전 통과 — 이미지 아닌 붙여넣기(텍스트·URL)는 그대로 통과 = 기존 텍스트 붙여넣기 무영향.

리스크·한계

  • 모바일 — Ctrl+V는 데스크톱 주 대상. 모바일 길게눌러 붙여넣기는 입력칸(textarea)에서 동작, 슬롯 자체는 브라우저별 차이 → textarea가 주 경로·슬롯은 폴백.
  • non-editable 슬롯 — div/label 자체 paste는 브라우저마다 안 올 수 있어 폴백으로만 의존, 주 경로는 짝 textarea·모달·document 위임(전부 실측 PASS).
  • 품질저하 리스크 — 기존 파일선택·삭제·미리보기 경로 전부 무접촉(read 경로 공용화만) = 회귀 0.
