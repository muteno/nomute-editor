①전송버튼 디밍 통일 — 라이브 비교

아래 버튼을 직접 눌러보면 게이지 채움→✓ 체크까지 실제 모션이 돈다. 평상시 바탕만 다르고 발사/체크는 셋 다 동일.

현재흰색 글래스 + 중립 테두리
애매한 디밍
background:rgba(255,255,255,.05)
border:var(--line) (흰 8%)
→ 바탕이 뿌옇게 떠 보임·강조 신호 약함
통일안 A.mkbtn 정본 계승
추천 · 반영함
background:rgba(0,0,0,.34)
border:rgba(15,253,2,.36) + blur
→ 제작('슛') 버튼과 같은 글래스-강조 언어
통일안 B.fb-save 톤
대안
background:rgba(15,253,2,.12)
border:rgba(15,253,2,.36)
→ 평상시에도 옅은 초록 존재감

↑ 버튼 클릭 = 발사 모션 재생(0.7초 게이지 → ✓ 팝). 다시 누르려면 잠시 후.

왜 A를 추천? 사용자가 가장 자주 보는 주요 액션 버튼이 .mkbtn(카드 '제작/슛')이고, 그 정본이 "검정 글래스 + 초록 테두리 + 네온 글자 + blur"다. 전송버튼을 여기에 맞추면 "최대한 기존이랑 맞춤"이 정확히 충족된다. 또 .fb-save(피드백 기록)도 원래 초록 테두리를 의도했어서, 흰색 글래스 .asksend가 사실상 유일한 이탈이었다. 검정 바탕은 초록 게이지 채움 대비도 흰색보다 더 또렷하다.

③입력 중 배경 깜빡임 — 원인 & 재발 방지

방아쇠60초 livePoll() + 1초 tickClock() + revise 완료 폴이 load()→render()로 배경 DOM(피드·#mdbody)을 다시 그림
증폭기수정 팝업은 backdrop-filter:blur() 모달. 뒤 DOM이 바뀌면 GPU가 블러를 재합성
증상타이핑 중 폴이 겹치는 순간 배경이 번쩍(간헐적·유지보수 때마다 재발)
수정isComposingModalOpen() 게이트 추가 → 작성 모달이 열려 있으면 livePoll·tickClock·revise/카드수정 완료 폴이 배경 재렌더를 보류(다음 틱·닫은 뒤 자동 따라잡음)
효과모달 뒤 DOM이 정지 → 블러 재합성 없음 → 깜빡임 0

⚠️ 필수 참조사항 — 재발 방지(이 기능 유지보수 시)

원칙: backdrop-filter(블러) 오버레이가 떠 있는 동안 그 뒤 DOM을 건드리지 말 것. 배경이 바뀌면 블러가 재합성되며 번쩍인다(Chromium/WebKit 공통 동작).

체크리스트: 새 폴링·타이머·자동갱신(setInterval/폴)을 추가하거나 render()/load() 호출 경로를 손볼 때 → 반드시 isComposingModalOpen()(또는 동급 가드)로 작성 모달 열림 중엔 배경 재렌더를 건너뛸 것. 새 작성 모달을 만들면 그 id를 이 가드 목록에 추가.

정본 위치: viewer/index.html isComposingModalOpen 주석 블록 + docs/작업이력.md §우선사항.

④완료 체크 픽토그램 일관성 — 확인 결과 ✅

전송버튼 완료 체크와 레포 내 다른 버튼들의 체크가 같은 픽토그램(path)인지 확인 → 동일 확정.

전부 path d="M5 13l4 4L19 7" · stroke:var(--on-accent) · stroke-width 2.7 · round cap/join 동일. 차이는 진입 모션뿐(전송=팝 스프링 ackPop / 제작=스트로크 드로우 goTick) — 이건 의도된 변형(260622 시안2). 픽토그램 자체는 통일됨 → 변경 불요.
