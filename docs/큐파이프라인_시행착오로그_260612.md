## 사건 1 — APK "앱이 설치되지 않음" (서명 불일치)

- **증상**: Termux:Widget(F-Droid판) 설치 시 "앱이 설치되지 않음". F-Droid 클라이언트 자체도 "알 수 없는 오류"로 설치 실패. Play 프로텍트 차단 해제로도 변화 없음.
- **처음 짚은 것**: 보안 설정/Play 프로텍트 의심 → 이미 꺼져 있어 기각.
- **실제 원인**: Termux **본체가 GitHub 릴리즈 debug 빌드**(서명 키 A)인데 부속 앱을 F-Droid 빌드(서명 키 B)로 설치 시도. Android는 같은 패밀리 앱의 서명 불일치를 설치 단계에서 거부하지만 에러 메시지에 사유를 안 적는다.
- **해결**: 부속 전부 GitHub 릴리즈 `+github.debug.apk`로 통일 — Widget v0.14.0, Termux:Tasker v0.9.0. curl로 받아 **내 파일 앱**에서 설치(termux-open은 무반응).
- **교훈**: ① Termux 패밀리는 **서명 출처 통일이 제1원칙** ② "앱이 설치되지 않음"류 무정보 에러는 서명부터 의심 ③ 본체 서명 확인법: `termux-api github.debug` 같은 패키지 버전 문자열.

## 사건 2 — Tasker "플러그인 오류: 누락, 비활성화, 내보내지지 않음 또는 권한 수신기 없음"

- **증상**: Tasker에서 Termux:Tasker 플러그인 선택 시 위 에러 토스트, 설정 화면 진입 불가.
- **처음 짚은 것**: 플러그인 미설치/Tasker 권한 — 절반만 정답.
- **실제 원인(핵심)**: `~/.termux/termux.properties`에 `# allow-external-apps = true`가 **주석 상태로만** 존재. 과거 다른 세팅 때 템플릿으로 들어와 있어 "켜져 있다"고 착각하기 딱 좋은 모양. 외부 앱의 Termux 명령 실행은 (a) 안드로이드 쪽 Tasker 권한 + (b) Termux 쪽 allow-external-apps **둘 다** 필요.
- **해결**: 주석 없는 실제 줄 추가 + `termux-reload-settings` → 에러 소멸, 설정 화면 정상 진입.
- **교훈**: 설정 파일은 **grep 결과의 `#` 유무까지** 본다. "값이 보인다"와 "켜져 있다"는 다르다.

## 사건 3 — 텍스트 공유가 Termux 저장 다이얼로그로 빠짐 (+ 파일명 ENOENT)

- **증상**: 기사 본문 블록 → 공유 → Termux 선택 시 큐 등록이 아니라 "Save file" 창. 제안 파일명에 URL 슬래시가 들어가 저장도 ENOENT로 실패. 파일명을 수동 수정(EDIT)해야만 겨우 통과.
- **분석**: Termux는 공유 텍스트가 **순수 URL로 시작할 때만** termux-url-opener를 호출하고, "포함된 링크: https://… 제목" 같은 텍스트 덩어리는 TermuxFileReceiver(저장창)로 보낸다. 이 수신기는 **앱 자바 코드에 컴파일된 부분**이라 스크립트/설정 레이어에서 못 고친다.
- **해결**: 텍스트 공유 동선을 **Tasker 경유로 우회** 설계(사건 4~5로 이어짐). 운영자 요구가 "블록 → 공유 → 수신처 탭 → 끝, 화면 그대로"였으므로 파일명 수동 수정 동선은 기각.
- **교훈**: ① 고칠 수 없는 계층(앱 컴파일 코드)에 부딪히면 **그 앞단에서 우회**가 정답 ② "동선 요구사항"이 아키텍처를 결정한다 — 같은 기능이라도 탭 수가 다르면 다른 설계다.
- (잔여 아이디어) Termux 앱 소스의 파일명 sanitize 패치는 별도 프로젝트로 보류.

## 사건 5 — 이벤트 검색 "공유" 0건 → 정답은 앱 카테고리의 "Received Share"

- **증상**: 프로필 → 이벤트에서 "공유" 검색 → 결과 없음.
- **원인**: 이벤트명이 한국어판에서도 영어 그대로 "Received Share"이고, 소속 카테고리가 직관과 달리 **앱**.
- **해결**: 카테고리 브라우징으로 발견(운영자가 직접 찾음) → 공식 문서로 변수 체계(`%rs_*`) 확인 → Task의 Arguments를 `%text`(잘못된 추정)에서 **`"%rs_text"`**로 교체. 따옴표는 공백 분할 방지, Termux 쪽 스크립트도 `"$1"` → **`"$*"`**로 보강(쪼개져 들어와도 전체 재결합).
- **결과**: 공유 → Tasker → "큐 등록됨" 알림, URL까지 표시. 입구 4종 완성.
- **교훈**: ① 현지화 앱에서 기능 검색은 **영문명 병행** ② 변수명은 설정 화면의 '출력 변수' 목록 = 1차 사료 ③ 외부에서 들어오는 문자열 인자는 **인용부호 + `$*`** 이중 방어가 싸게 먹힌다.

## 사건 6 — Cloudflare "Pages가 사라졌다" (Workers 강제 유도)

- **증상**: Create application을 누르면 Pages가 아니라 **Worker 생성 플로우**(Deploy command `npx wrangler deploy`)만 나옴. 운영자: "저번에도 여기서 애먹었던 것 같은데."
- **오진**: 🤖A가 "생성 화면의 Pages 탭을 찾아라"라고 안내 → 그런 탭이 사실상 없어진 상태(낡은 정보, 정정함). 계정 문제/다른 계정 배포 의심도 있었으나 기존 홈페이지(nomute)가 같은 계정에 멀쩡히 존재 — 기각.
- **실제 원인**: Cloudflare가 2025년 말부터 신규 Pages 생성 UI를 숨기고 Workers(Static Assets)로 유도 중. /pages 경로가 리다이렉트되는 사례가 공식 커뮤니티에 보고됨.
- **해결**: **딥링크 직접 입력**이 생존해 있었음 →
  `dash.cloudflare.com/<ACCOUNT_ID>/pages/new/provider/github`
  → 정상 Pages 생성 화면 진입 → 설정값: branch `main` / preset `None` / build `node build-viewer.mjs` / output `viewer` → `nomute-editor.pages.dev` 확보.
- **교훈**: ① 플랫폼이 기능을 "숨겨도" 딥링크는 한동안 산다 — URL 패턴을 기록해 둘 가치 ② 첫 "Success"는 **프로젝트 생성** 성공이지 빌드 성공이 아님(머지 전이라 첫 빌드 빨간 X는 예정된 정상) ③ 재구축 시점엔 Workers Static Assets(wrangler.jsonc + assets.directory)가 정석 경로일 수 있음 — 그땐 그쪽으로.

## 사건 7 — Run #1 전멸: 401 Invalid bearer token (오판과 분리 검증)

- **증상**: 첫 가동(run #1) 36초 만에 종료, pending 6건 전부 `failed/`행, `.log`마다 `401 Invalid bearer token`. (워크플로 상태는 Success — 격리 설계가 "정상 수행"됐기 때문. **초록 ≠ 분석 성공**임을 처음 배운 지점.)
- **오진 1 (🤖E)**: "토큰 값이 문제 ✅확정 — 만료/잘림/API 키 오입력" → 재발급 지시.
- **운영자 반론 (👤)**: 시크릿창으로 계정 분리해 발급, 두 토큰 값 상이 확인 — 절차 정상이라는 항변. 추가 의심: "CLI가 계정별로 고유해야 하나?" "이름 대소문자 탓 아닌가?"
- **분리 검증 (🤖A 제안, 👤 실행)**: 재발급 **전에** 로컬 생사 판정 —
  `CLAUDE_CODE_OAUTH_TOKEN="sk-ant-oat01-…" claude -p "ping"` → **pong = 토큰 무죄 확정.** 🤖E의 "확정" 진단 기각.
- **사실 정리(검증 기반)**:
  - setup-token 토큰은 **브라우저 로그인 계정**에 묶인다. 같은 PC·같은 CLI로 여러 계정 발급은 정상 패턴(CI용으로 설계된 기능).
  - GitHub 시크릿 **이름은 대소문자 무시 + 저장 시 대문자 정규화**(공식 문서) — 이름 케이스로는 401이 날 수 없음.
  - 401의 진짜 후보 = 러너 쪽: env 변수명이 `CLAUDE_CODE_OAUTH_TOKEN`이 아니거나, `ANTHROPIC_API_KEY`/`ANTHROPIC_AUTH_TOKEN`이 OAuth를 조용히 덮어쓰는 알려진 이슈(anthropics/claude-code#34826).
- **조치**: 만약을 위해 시크릿 2개를 CLI 출력 직복붙으로 재등록 + 워크플로에 3중 방어 추가(빈 시크릿 즉시 에러 / 길이+접두사 12자 디버그 / ANTHROPIC_* unset). → run #2에서 `token len=108 prefix=sk-ant-oat01` 로그로 인증 통과 확인. (진범이 '잘림'이었는지 'env 간섭'이었는지는 동시 수정이라 단정 불가 — 둘 다 막아놨다는 게 결론.)
- **교훈**: ① **재발급 전에 분리 검증** — 변인 하나짜리 생사 테스트가 가장 싼 진단 ② "✅확정"이라는 말과 실제 확정은 다르다, AI의 단정도 검증 대상 ③ 시크릿은 재열람 불가(쓰기 전용)이므로 "뭘 넣었더라" 싶으면 고민 말고 덮어쓰기 ④ 디버그는 값 노출 없이 **길이+비민감 접두사**만 찍는다.
- **📌 기록 차이 주의**: 에디터의 최종 보고서(`큐레이션자동화_구축보고_260612.md`)는 이 건의 원인을 "등록 토큰 값 무효(만료/오타) → 사용자 토큰 재발급"으로 기록했으나 **실측과 다르다** — 토큰은 로컬 ping 검증으로 유효가 입증됐고, 재발급 없이 동일 토큰 재등록 + env 방어를 동시 적용했다(따라서 진범은 '시크릿 저장값 vs env 간섭' 중 단정 불가). 교본에는 **본 로그가 정본**.

## 사건 8 — Run #2 사망: 한글 파일명 멀티바이트 절단 (Opus 산출물 증발)

- **증상**: run #2가 6분 34초 돌다 **Failure(exit 1)**. 로그: `ENOENT … 'queue/260612-2324-안민석-개표입력오류-전문가�.md'` — build-viewer.mjs에서 사망.
- **오진 (👤)**: "내가 따로 돌리던 작업이랑 충돌/쿼터 탓?" → 로그로 기각(쿼터·push 충돌 흔적 없음).
- **실제 원인**: 분석 스텝의 **AI(Opus 4.8)가 자동 작성한 파일명 슬러그 로직**(`cut -c` — 문자가 아니라 **바이트** 단위 절단)이 md 파일명에 한글 제목을 넣고 길이를 자르다 UTF-8 멀티바이트 문자 **중간**을 절단 → 깨진 바이트(`�`)가 든 파일명 생성 → build-viewer의 readdir↔readFileSync 왕복 불일치 → ENOENT. 빌드가 죽으면서 **push 스텝 도달 전 산출물이 러너와 함께 증발** — 6분치 분석 손실.
- **역설적 수확**: 죽은 파일명이 곧 "네이트 실기사 분석 성공"의 증거(403 우려를 뚫고 제목까지 뽑았음). 인증·분석 라인은 이때 이미 완주.
- **해결(구조 반영, 리포 main 실측 확인)**: ① md 파일명 **ASCII 한정**(타임스탬프+기사ID), 한글 제목은 frontmatter `title`로 ② **결과 커밋을 빌드보다 선행 + `if: always`** — 뭐가 죽어도 분석물은 보존 ③ viewer 빌드를 워크플로에서 제거, **Pages 전담**으로 책임 분리 ④ build-viewer에 못 여는 파일 skip 방어.
- **교훈**: ① **파일명은 데이터가 아니다** — 의미(제목)는 메타데이터로, 파일명은 ASCII 식별자로 ② 비싼 산출물(모델 출력)은 **만들어진 즉시 영속화**, 후처리는 그 다음 ③ AI가 짠 코드의 단골 사각 = 멀티바이트/인코딩 경계 — 한국어 파이프라인에선 1순위 리뷰 항목 ④ 실패의 부산물(깨진 파일명)도 읽어라, 거기 성공의 증거가 들어 있었다.

## 횡단 교훈 — 교본에 올릴 원칙들

1. **분리 검증이 왕이다.** 토큰 401(사건 7)도, "내 작업 탓?"(사건 8)도, 변인 하나짜리 테스트/로그 한 줄이 추측 전체를 정리했다. 재발급·재설치 같은 비싼 조치는 항상 검증 뒤에.
2. **초록불의 의미를 정의하라.** 격리 설계에선 전멸도 Success(사건 7), 빌드 사망은 Failure(사건 8). 상태색이 아니라 **소요시간·산출물·로그**가 진짜 신호다 (36초=즉사, 6분=실분석).
3. **비싼 것부터 영속화.** 모델 산출물 > 빌드 > 배포. 순서를 이 우선순위로 박아야 하위 단계 실패가 상위 가치를 못 태운다.
4. **고칠 수 없는 계층은 우회한다.** 앱 컴파일 코드(사건 3), 플랫폼 UI 정책(사건 6) — 싸우지 말고 앞단에서 갈아탄다. 단 딥링크·구버전 경로 같은 "아직 살아있는 문"을 먼저 두드려 볼 것.
5. **이름·인코딩·인용부호는 공짜 보험.** ASCII 파일명, `"%rs_text"`의 따옴표, `"$*"` — 전부 한 글자급 비용으로 사건 하나씩을 막는다. 한국어 파이프라인이면 멀티바이트 경계는 디폴트 의심 지점.
6. **설정은 '존재'가 아니라 '활성'을 확인.** 주석 처리된 true(사건 2), 빈 Environment secrets 박스(무해) — 보이는 것과 켜진 것을 구분.
7. **AI 협업의 규율**: AI의 "확정"도 검증 대상(사건 7) · AI가 짠 코드의 인코딩 경계는 사람이 본다(사건 8) · 단, 방어 코드 3종 세트(빈 값 가드·비민감 디버그·env unset)처럼 AI가 선반영한 모범도 있다 — 신뢰하되 검증.
8. **자격증명 위생**: 시크릿은 쓰기 전용 → 의심되면 덮어쓰기 / 디버그는 길이+접두사만 / PAT는 최소 권한(리포 1개·Contents만)+만료 캘린더 / 토큰 원본은 패스워드 매니저가 정본.
9. **의심스러우면 실패가 안전한 기본값.** 사실 무결성이 우선인 시스템에서 "거부"는 비용이 아니라 기능이다 — 환각 제목을 스스로 기각한 n12345 건(사건 11)이 증명. 성공률을 올리려고 거부 문턱을 낮추는 순간 날조가 들어온다.
10. **LLM→기계 인터페이스는 3중 방어**(사건 10): 후처리 절단 · 관용 파서 · 프롬프트 명시. 그리고 기록이 둘이면(사건 7의 보고서 충돌) **실측 로그 쪽을 정본으로 지정**해 두라 — 교본의 신뢰는 거기서 갈린다.
