# 수집함 큐레이션 알고리즘 — 설계 SSOT (살아있는 문서)

> 이 문서 = 수집함(누적/신속) 분류·정렬 알고리즘의 **설계 정본.** 세션이 압축되거나 끊겨도 여기 박혀 있어 안 날아간다. 결정될 때마다 갱신.
> 코드 정본: 분류·정렬 = `viewer/index.html`(`renderScrap`·`scScore`·`scFast`) / 신호 산출 = `scraper/to_candidates.py` / 운영자 사유 라벨 = `docs/curation-rubric.md`.

## 0. 목적 (왜 이 앱이 특별한가)
내용·키워드 **없이**, 오직 **보도가 퍼지는 모양(템포)**만으로 "진짜 터진 사건(사고·재난)"과 "돈 받고 동시 배포한 보도자료"를 가른다. 시중 앱엔 없고 보통 유료. 운영자가 직접 개발.

## 1. 사용자 요구 (정본)
- **누적 = 9시 뉴스감**: 다수 매체 일제 보도·전국민 공감·**화두 큰 것부터** 오프닝.
- **신속 = 속보 캐처**: 방송 중 하단자막 같은 대형 화재·사고·재난. 예정 이슈(수상+글로벌스타)도 신속 가치.
- **평가축 = 보도 템포.** 다수 매체 보도 *이상으로* 중요한 게 **연속 보도**(수집할 때마다 또 올라옴 = 전국민 대형급). **여기에 가점이 제일 무겁게.**
- 내용·키워드로 못 가르는 **"예정-중요"만 키워드(고유명사)**로 건짐.
- **메이저 매체 가점 없이도** 부상.
- **별점 피드백(~30일)**으로 운영자 취향 보정 = 최종 정확도(운영자가 2차 큐레이션 하니 1차는 근사면 충분).
- 중요한 게 묻히면 안 됨. 더 중요 → 더 먼저. 신속 6h도 길 수 있음.
- ⛔ **발행시각(정각·업무시간)은 신호로 안 씀** (운영자 지시 · 260616).

## 2. 핵심 원리 — 내용 없이 가르는 법
| | 보도자료(돈/홍보) | 사고·사건(진짜) |
|---|---|---|
| 퍼지는 모양 | **동시 분사 → 즉사** | **선두 1곳 → 시차 확산 → 계속 자람** |
| burst vs cross | burst ≈ cross (다 한 번에) | burst < cross (시차로 번짐) |
| 성장 꼬리(`last_seen − first_seen`) | ≈ 0 | 큼 |
| 후속 | 없음(한 번 깔고 죽음) | 이어짐(상보·속보) |
| 처리 | 신속에 잠깐 떴다 **곧장 강등** | 신속·누적 상위 유지 |

→ 🟡 **실험(260616 candidates 754건 실측): 구조 신호(cross·꼬리)만으론 보도자료 vs 진짜 사건 못 가름.** cross 21에 '미·이란 종전'과 '현대차 홍보' 공존 · 꼬리도 안 갈림(보도자료 '청정원 홍초 출시' 6.8h vs 사건 '미·이란 종전' 0h · 평균 꼬리 사건 2.68h ≈ 보도자료 2.04h).
→ **결론: 순수 코드(키워드·패턴)로 못 푼다 → AI(Claude)를 사람 대신 검증자로 끼운다 (§2.5).**

## 2.5 ★핵심 아키텍처 (260616) — 일반 알고리즘(코드) + 특이케이스(AI 검증)
순수 코드로 다 풀면 키워드·패턴 지옥. **AI를 게이트에 끼우면 훨씬 단순** — 구조 신호가 *후보만* 추리고, 애매한 결정은 Claude가 내용 읽어 *사람 대신* 경중 판정.

**① 일반 알고리즘 (코드·무료·항상 작동):** 구조 신호로 누적/신속 분류·정렬.
다수매체(cross) → **연속 보도 가점** → 최신순 최상단 → 임계 넘으면 알람 후보.

**② 특이 케이스 핸들러 (AI·타겟 과금·따로 만듦):** 코드가 못 가르는 것만 Claude가 내용 1콜로 경중 판정.
- **케이스 A — 빠른 단독 특종**: `[단독]` 헤더 + 단일 매체 1건만 수집 → cross 낮아 일반 알고리즘선 묻힘 → **Claude가 깐다**(언론고시 본 기자라 `[단독]` 헛소리 안 씀 = 값어치). 디스패치류 연예특종은 출처로 라우팅 가능.
- **케이스 B — 다수 매체 동시 분사**: 한 시점에 다수가 쫙 보도 → 보도자료(홍보)냐 누가 봐도 엄중한 사안이냐 구조론 모름 → **Claude가 내용 읽어 판정 → 엄중하면 알람.**

**이미 절반 있음**: `breaking_judge.py`(burst≥3 → Claude가 사고/재난 판정) = 정확히 이 패턴. **확장만** 하면 됨(처음부터 안 짬).

✅ **구현됨(260616)** = `.github/scripts/gate_judge.py`: 노출 후보(cross≥3 or 속보후보 or `[단독]`)를 Claude가 **0~3 경중 채점**(0 비뉴스/보도자료 · 1 경미 · 2 주목 · 3 대형). RUBRIC 해시 도장(`grade_rubric` = 드리프트 차단)·청크 120·self-gate(`--count`)·`breaking-judge.yml` 동거(구독 OAuth 재사용). 뷰어 `gradeW`(0→×0.05 침몰 · 3→×1.8 부상 · 미채점→×1 중립)를 신속·누적 점수에 곱 + 알람 grade≥2 게이트. 구조 신호(AUC 0.5~0.59) 한계를 AI 내용판정이 보완.

## 3. 신호 (전부 내용·키워드 0 · 현 데이터)
- `cross` = 보도 매체 수
- `burst` = 15분 내 동시 보도 매체 최대 (동시성)
- `first_seen` / `last_seen` = 처음 본 시각 / **마지막으로 cross가 자란** 시각
- **성장 꼬리 = `last_seen − first_seen`** = 얼마나 오래 새 매체가 계속 붙었나 (= 연속 보도)
- `seen_count` = 수집 사이클마다 등장 횟수. ⚠️ **현재 결함**: 정체 등장(보도자료가 피드에 그냥 남아있는 것)도 +1 → 보도자료 오가점. **개선 = `grow_count`(cross가 *실제로 자란* 사이클 수)로 바꿔야 연속 보도를 정확히 잼.**

## 4. 신속 (속보 캐처) — 규칙 (260616 운영자 확정)
- **피드 진입**: `cross≥2` (2개 이상 매체가 같은 내용 보도).
- **속보 알람**: `cross≥4` (스크랩 시 4개 이상 매체). **단, 순수 cross 알람은 89% 오발령(§7.5) → AI 게이트(§2.5 케이스 B) 통과분만 실제 알람.**
- **하강(피드서 내림)**: 후속 보도 있나? → **후속 있으면 가점=유지, 없으면 계속 하강**(한 번 깔고 죽는 보도자료 = 즉시 침몰).
- 🔴 **현재 한계(실험 §7.5)**: "후속"을 `last_seen−first_seen`(꼬리)으로 재면 **틀림** — 큰 사건은 우리가 보기 전 이미 만렙이라 꼬리 0 → "후속 없음" 오판(미·이란 종전·추락사망이 하강 대상으로 잘못 찍힘). **꼬리 = '사건 지속'이 아니라 '우리가 일찍 잡았나'를 잼.** 진짜 후속 = **매 스크랩 cross 시계열 기록**해야 정확.

## 5. 누적 (9시뉴스) — 초안
- 점수 = `log2(cross)` × **연속보도 가중(성장 꼬리/grow_count — 제일 무겁게)** × 후속 신선도 감쇠.
- 화두 큰 것 = 넓게 + 오래 + 계속 보도된 것. 보도자료(성장 0)는 바닥.

## 6. 별점 보정 (phase 4)
알고리즘 = 근사치. 운영자 별점/트리아지 ~30일 누적 → 취향 반영 가중. 1차(신호)가 근사면 운영자 2차 큐레이션이 마무리.

## 7.5 실험 결과 (260616 · 오늘 24h 742건 실측)
- **cross≥4 알람 = 213건 발령, 89%(191건)가 사건 아님**(보도자료·부고·잡음). 사건은 22건뿐. → 순수 cross 규칙 = 알람 폭주·오발령 → **AI 게이트(§2.5) 필요 입증.**
- cross≥5 라벨별 평균 꼬리: 사건 2.68h ≈ 보도자료 2.04h = **꼬리로 안 갈림.**
- **꼬리(후속) 신호 오작동**: 최대 사건(미·이란 종전 21·추락사망 24)이 꼬리 0 → 하강 대상 오분류. 원인 = 꼬리가 관측 타이밍 의존(§4 한계).
- 결론: 구조 신호 = *후보 추리기*까지만, 경중 판정 = AI. 진짜 후속 측정엔 **cross 시계열 기록** 필요.

## 9. 추후 설계 — 데이터 누적 + 미래형 검증 (260616) ★로드맵
전제: **첫날부터 완벽 불가.** 데이터가 쌓여야 패턴·특이케이스가 드러난다(오늘은 특이/희귀가 적을 수도). 그래서 *지금부터 올바른 데이터를 누적*하고 검증을 *반복 가능*하게 짜둔다.

### A. 데이터 누적 층 (모든 것의 토대 — 안 모으면 영영 사라짐)
1. **주기적 스냅샷 아카이브**: `candidates.json` → `scraper/snapshots/{날짜}.json` 보존(live는 10일 후 폐기). *과거 데이터 = 검증·재현의 원료.* (현재 git 이력에 매 스크랩 candidates가 커밋되긴 함 = 거친 스냅샷 — 깔끔한 전용 아카이브가 나음.)
2. **사건별 cross 시계열**: 스냅샷 교차로 각 사건 cross 궤적 복원 → **진짜 후속(꼬리) 측정**(현 꼬리 결함 §7.5 해소). 핵심.
3. **라벨 누적**: `scraper/labels.json`(AI/수동 정답) 계속 추가 → 검증 신뢰도↑.

### B. 로직 (데이터 쌓일수록 좋아지게)
- **2층 구조**(§2.5) = 코드 게이트(싼 후보 추리기) + AI 검증(경중). 첫날부터 작동, 데이터 무관.
- **임계값 = 고정 아님**: 데이터 쌓이면 `hypothesis_lab`이 최적 임계 재계산(cross≥N, 후속 가중…).
- **특이케이스 핸들러 = 플러그인식**: 새 패턴 발견 = 케이스 추가([단독]·동시분사·…). 무한 확장.
- **별점 30일 보정**(phase 4) = 운영자 취향 최종 수렴.

### C. 검증 프레임 (미래형 — 반복·추적)
- **랩(`scraper/hypothesis_lab.py`)**: 신호 분리력(AUC)·알람 정밀도. 새 가설 = `SIGNALS` 한 줄.
- **시계열 추적**: 스냅샷마다 랩 돌려 *지표가 시간 따라 개선되나* 추적(데이터·라벨 쌓이며 AUC·정밀도 변화 = 알고리즘이 학습되는지 객관 측정).
- **AI 라벨링 루프**: Claude 배치 라벨 → `labels.json` → 정답 정제 → 랩 신뢰↑. **(이게 곧 production AI 게이트와 동일 메커니즘 — 한 번에 두 마리.)**
