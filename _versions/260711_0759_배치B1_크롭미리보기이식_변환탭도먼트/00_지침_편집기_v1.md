## 2. 캡·예산 (기틀 — 완화 = 운영자 확인)

## 3. 알려진 한계 (정직 · v1)
- a) 트림×자막 동시 = 지원(260711 리맵) — 단 STT·의역은 **전체 영상** 기준으로 돌고 번인만 구간 반영(구간 밖 의역 = 비용 낭비 소폭 · 트림 선반영 STT는 후속).
- b) 컷 편집 자막 의존 = 해제(260711 STT-only — §1-b). 잔여 한계 = 컷 단독도 STT(전사) 비용은 든다(Whisper 수 분) · 전사 실패 시 컷 스킵 note(다른 처리 진행) · 컷 *단독* 스택 + 전사 실패 = 정직 에러(산출할 게 없음).
- c) 다시 입히기(reburn)·상세 자막 편집기·자막 텍스트 뷰 = 편집기 v1 미탑재(자막 생성기 탭 그대로 사용 — 같은 `ly_out/<id>` 계약이라 후속 접합 쉬움). **재입히기 편집 설정 승계(260711 후보7)**: 편집기 산출 video.json에 `edit_opts` 스냅샷(vid_*·aud_norm) 도장 → reburn(자막 다시 굽기) 시 ly_burn이 자동 병합 = 여백·해상도·보간·음량·트림 유지("지금은 자막만" 소실 봉합 · 이번 opts에 편집기 축 명시 = 병합 안 함 · note "이전 편집 설정 승계" 표면화 · ly.html/api/ly·ly-make 무접촉 = ly_burn 단독 완결).
- d) 크롭 위치 미리보기(드래그 프레임) = v1 미탑재(위치 슬라이더만 · conv 탭엔 있음 — 후속 이식).
- e) 컷 + 60fps 보간 동시 = 컷 이음매에서 보간 워블 가능(minterpolate 장면전환 감지가 완화 · 원리 한계).
- f) 예상 소요 = 근사(60fps는 파일이면 길이 기반·URL이면 중간값) — 실측은 대기 캡션 경과 초.
- g) **클리퍼 v0 한계**: 후보 확정 후 렌더는 새 잡 = 자막 ON이면 STT 재실행(스캔 segments.json 재사용 = 후속 최적화) · 원본 보관 400MB 캡(초과 = 파일 재선택) · 입력 캡 = 편집기 공통 10분(롱폼 완화 = 운영자 확인 필요 기틀).
